# 数据库备份/恢复

在 PostgreSQL 中，备份与恢复（Backup and Restore）是保障数据安全的核心操作。

根据业务场景的不同，主要分为三种方式：**逻辑备份、物理备份（冷备份）** 和 **连续归档/物理热备份（PITR）**。

| **备份类型**                   | **优点**                                      | **缺点**                                     | **适用场景**                                 |
| ------------------------------ | --------------------------------------------- | -------------------------------------------- | -------------------------------------------- |
| **逻辑备份 (`pg_dump`)**       | 跨版本兼容性好；可单独恢复某张表；文件可读。  | 备份和恢复速度慢（大数据库不适用）。         | 数据量小（< 50GB）；需要导出数据到开发环境。 |
| **物理备份 (`pg_basebackup`)** | 备份和恢复极快。                              | 不能跨大版本恢复；无法选择性恢复单张表。     | 大型数据库的日常例行全量备份。               |
| **PITR (连续归档)**            | 可恢复至任意时间点；RPO（数据丢失量）接近零。 | 配置较复杂；需要占用较大的存储空间存储日志。 | 核心生产环境；对数据安全性要求极高的业务。   |

## 基本概念

​	备份：指的是**创建数据库副本的过程**，它确保了数据的**持久性和连续性**，以便在发生数据丢失、损坏、硬件故障或意外删除时可以恢复。

### 核心优势

- **数据还原**：数据恢复能够在因硬件故障、软件错误或人为错误导致数据丢失后进行数据还原
- **灾难恢复**：在发生灾难性故障（如系统崩溃、勒索软件攻击、意外删除）时至关重要
- **迁移支持**：有助于将数据从一台服务器迁移到另一外服务器上，或在升级 PostgreSQL 版本时使用
- **测试与开发**：备份文件可以在测试环境中恢复，以模拟生产环境问题或测试新功能
- **时间点恢复（PITR）**：允许将数据库恢复到特定时刻，这对于从逻辑中恢复非常有用
- **合规性与审计**：定期备份有助于满足关于数据保留和恢复的法律及法规要求

## 一、逻辑备份与恢复

逻辑备份是将表结构和数据导出为 **SQL 语句**或**特定格式的文件**。这种方式灵活性高，适合中小型数据库。

### 单个数据库备份

主要是通过 PostgreSQL 内置的 `pg_dump` 工具在 psql 命令行工具中实现：

#### `pg_dump` 备份

注意：`pg_dump` 命令是系统性工具，必须在 Linux 环境下运行。

> 备份的文件会存储在**当前文件夹**下。

- **备份为 SQL 文件（`.sql` 文本格式）**：

  ```sh
  $pg_dump -U [用户名] -d [备份数据库] > [数据库备份文件名].sql
  
  $pg_dump -U [用户名] -d [备份数据库] -v -f [数据库备份文件名].sql # -v 查看运行日志
  ```

- **备份为压缩自定义格式**（如 `.dump`）

  ```sh
  $pg_dump -U [用户名] -Fc -d [备份数据库] > [数据库备份文件名].dump
  ```

  - **`-F`**: 指定格式。`-Fc` 会生成自定义二进制压缩格式，体积更小，且支持并行恢复（推荐！）

- **仅备份模式、表结构**：

  ```sh
  $pg_dump -U [用户名] -Fc -d [备份数据库] -s [模式名] > [数据库备份文件名].dump
  
  $pg_dump -U [用户名] -Fc -d [备份数据库] -t [表名] > [数据库备份文件名].dump
  ```

  - **`-t`**: 只备份某张表（例如：`-t cloud_files`）
  - **`-n`**: 只备份某个模式（例如：`-n storage`）

#### `psql` / `pg_restore` 恢复

> 注：需要一定**权限**，常见于 `postgres` 超级管理员。

- **恢复 SQL 脚本（仅对 `.sql` 文本文件）**：

  ```sh
  $psql -U [用户名] -d [备份数据库] < [数据库备份文件名].sql
  
  $psql -U [用户名] -d [备份数据库] -f [数据库备份文件名].sql
  ```

- **恢复自定格式（对应 `.dump`）**：

  ```sh
  $pg_restore -U [用户名] -d [备份数据库] -v [数据库备份文件名].dump
  ```

  如果要利用多核加速恢复，可以使用 `-j` 参数（例如 `-j 4` 表示 4 线程并行恢复）。

### 全库备份与恢复

用于**备份整个 PostgreSQL 实例**，包括**所有数据库、角色（用户）和表空间**。常见于**数据库集群**场景。

- **`pg_dumpall` 备份全库**：

  ```sh
  $pg_dumpall -U [用户名] > [数据库备份文件名].sql
  
  $pg_dumpall -U [用户名] -f [数据库备份文件名].sql
  ```

- **`psql` 恢复全库**：

  ```sh
  $psql -U [用户名] < [数据库备份文件名].sql
  
  $set PGPASSWORD=root   # 不然就会导一个数据库输一次密码
  $psql -U [用户名] -f [数据库备份文件名].sql [要连接的数据库名]
  ```

  - **`-f`**: 代表 file，即指定要运行的 SQL 脚本文件。
  - **`postgres` (最后的参数)**：这是连接的初始数据库。因为`pg_dumpall`脚本内部包含了`CREATE DATABASE`和`\c`(切换库) 的命令，所以你只需要先连上一个默认存在的库（比如`postgres`），它会自动帮你创建并切换到其他的库。

## 二、 物理备份与恢复（适合大型数据库）

物理备份是**直接复制底层的数据库文件（如数据块、WAL 日志等）**。速度极快，适合数据量巨大的场景。

### 1. 冷备份（停机备份）

最简单粗暴的方法。

1. 停止 PostgreSQL 服务：`sudo systemctl stop postgresql`
2. 复制整个数据目录（通常是 `/var/lib/postgresql/data`）到备份位置。
3. 启动服务：`sudo systemctl start postgresql`

### 2. 热备份 (`pg_basebackup`)

无需停机，在数据库运行的同时进行文件级备份。

- **基础备份命令：**

  ```sh
  $pg_basebackup -U username -D /path/to/backup_dir -F p -P -v
  ```

  - **`-D`: 备份输出目录**
  - **`-F p`: 输出为原始格式（也可以用 `-F t` 输出为 tar 包）**
  - **`-P`: 显示进度**

## 三、 连续归档与基于时间点的恢复 (PITR)

如果业务要求 **“数据零丢失”** 或者需要 **“恢复到昨天下午 3 点 15 分的任意状态”**，就需要使用 PITR（Point-in-Time Recovery）。

### 1. 核心原理

通过**开启预写日志（WAL）归档**，将数据库的增量变化实时保存下来。恢复时，先还原一个历史的 `pg_basebackup` 基础物理备份，然后像“重放录像”一样，播放 WAL 日志到指定的精确时间点。

### 2. 配置步骤简述

1. 修改 `postgresql.conf`，开启归档模式：

   Ini, TOML

   ```
   wal_level = replica
   archive_mode = on
   archive_command = 'cp %p /path/to/archive_dir/%f'
   ```

2. 重启数据库并制作一个基础备份（Base Backup）。

3. 当发生灾难时：

   - 停止数据库
   - 清空数据目录，将基础备份拷回
   - 在数据目录下创建 `signal` 文件（如 `recovery.signal`）
   - 在 `postgresql.conf` 中配置 `restore_command`（告诉 pg 去哪里读归档日志）以及 `recovery_target_time`（目标时间点）
   - 启动数据库，它会自动重放日志直至指定时间
