# FastAPI 框架

> [!NOTE]
>
> 在 FastAPI 框架出现之前，Python Web 开发圈主要是由两大阵营所主导：
>
> - **Django**：自带大而全功能的**集成全家桶**，但非常**笨重**
> - **Flask**：**轻量且灵活性高**的**小型 Web 框架**，但需要手动组装各类插件
>
> 随着现在 Web 开发对**高并发、异步输入输出、前后端分离（纯 API 驱动）**的需求日益剧增，传统的**同步框架（Django、Flask）**逐渐暴露出**性能瓶颈**；在这个背景下，FastAPI 应运而生。
>
> - **FastAPI** 是一个**现代、快速（高性能）**的 **Web 异步框架**，用于构建基于 Python 3.8+ 与 使用了**标准 Python 类型提示（Type Hints）**的 **API 微服务**。
>
> > *`FastAPI` 与 Node.js 的 `Express/Koa2` 框架设计思想类似*：通过 **“非阻塞 I/O”**和 **“事件循环”**，单线程也能跑出极高的性能。
> >
> > ​	它们都是为了**高效率构建 Web 服务（尤其是 RESTful API）**而设计的**轻量级、富有现代感的 Web 框架**，可通过 **`async/await` 异步编程**，让**「单线程」**在面对**高并发I/O 操作**任务的业务场景下，能**通过「多个协程」借助事件循环（Event Loop）机制调度**，**在多个等待间的 I/O 任务之间快速穿梭**，以实现**高吞吐量**的效果。

## 介绍

FastAPI 是一个**高性能**的 **Web 异步框架**，用于**构建基于 Python** 的 **API 接口层服务**。

中文官网：https://fastapi.tiangolo.com/zh/

核心概念：

- 基于**标准的 Python 类型提示（Type Hints）**：定义**函数入参、返回值，变量**的**类型校验**

- 通过**利用 Python 的 `async/await` 异步编程（多协程）特性**，提供了极高的**并发**性能

- 基于 **`Uvicorn` 运行**的 FastAPI 程序是**最快的 Python Web 框架**

  > **`Uvicorn`** 是一个支持**异步**的 **Web 服务器**，可以通过**单点部署 FastAPI 程序**，实现高性能的服务。

- **支持 WebSocket 协议**，可以轻松地扩展到更复杂的应用场景

### 核心特点

- **高性能的异步并发**：

  ​	FastAPI 是**基于 `Starlette` 底层框架**之上的**上层应用框架**，使得 FastAPI 的**性能接近于 Golang** 和 **Node.js** 语言，可以实现比 Flask 和 Django 等传统框架更快的性能；

  ​	同时，FastAPI 的 **`async/await` 异步支持**使其能处理大量**并发**请求，并在**高负载**下**保持稳定**的性能。

- **自动生成接口文档（Swagger UI、Redoc）**：

  ​	FastAPI 使用 OpenAPI 标准，可通过 Swagger UI 或 Redoc **自动为每个 API 接口生成一个交互式 API 文档**，以此帮助开发者方便理解和使用 API 接口。

- **运用类型提示与数据模型的校验处理（Type Hints、Pydantic）**：

  ​	FastAPI 中完全采用 Python 3.7+ 的**类型提示（Type Hints）**，使得代码更简洁、清晰；同时还允许**采用 `Pydantic` 库**对**请求/响应数据**实现针对性的**数据模型（DataModel）的定义与自动验证、 JSON 序列化处理、错误处理**，使得开发者能轻松定义和验证复杂的数据结构（自动生成类型文档、类型自动转换...），**确保 API 函数的输入输出的数据结构 Data Schema 都符合预期**。	

- **`async/await` 异步支持（多协程）**：

  ​	FastAPI **原生支持 Python 的异步编程特性（`await`、 `await`）**，非常适合处理 **I/O 密集型任务**。如数据库查询、文件操作、外部 API 对接调用...

- **开发效率高**：

  ​	FastAPI 能帮助开发者在更短的时间内开发出性能优秀、类型安全的 API 接口。还额外提供了许多内置工具，如**数据验证、序列化、依赖注入**等，减少了对样板代码的需求。

  ​	此外，FastAPI 还提供了一些常见的功能，如 **JWT 身份验证、ORM 数据库集成**，以便开发人员可以快速地构建功能完整的 API。

### 与 Django、Flask 的区别

**`FastAPI`、`Flask`、`Django`** 都是**基于 Python 语言实现的 Web 框架**，但它们的核心特征与功能关注点有所不同：

- **FastAPI**：主要**面向于 API 接口层应用**，非常适合**构建高性能的纯 API 微服务**。特别是在需要处理**大量并发请求**或**数据验证**的业务场景（**机器学习、数据分析**...）
- **Flask**：一个**轻量级的 Web 应用框架**，适合用于**简单的 Web 应用或原型开发**。但需要**额外手动配置一些插件来处理复杂的请求**
- **Django**：一个**全功能的 Web 应用框架（全家桶）**，适合用于**构建一个包含数据库、JWT 权限验证、管理后台的复杂 Web 应用**，但对于纯 API 的微服务项目来说就过于**笨重**了

> 一般来说，**FastAPI 是完全可以替代 Flask** 的，所以基本上就是 FastAPI 和 Django 二者选其一。

### 开发特点

1. 使用了大量的 Python 类型声明，即从 **`from typing` 模块中引入大量数据类型**

   ​	**函数的入参都会声明类型**，方便数据**类型检查、代码提示**

2. 大量使用**数据模型类**，例如 API 的入参、处理、返回、到数据库的存储；

   ​	例如 **API 函数的入参 和 返回值，都预定了一个数据模型类**，表示**数据的 Schema 结构（有哪些字段、字段的类型）**

   *类似于 Java 中 SpringMVC 的 Bean 实体类*

3. 对于数据库、API、文件的调用，**大量结合 `async/await` 异步编程**的写法

### 为什么 FastAPI 能这么快？

#### 程序提速的方式

- **多进程（`multiprocessing` 模块）**：利用**多核 CPU** 能力，真正的**并行执行任务**
- **多线程（`threading` 模块）【实际还是单线程---GIL “锁”】**：利用 **CPU 和 IO 可以同时执行**的原理，让 **CPU 不会傻傻等待 IO 完成**
- **多协程（`asyncio` 模块）【异步编程】**：在**单线程**中**利用 CPU 和 IO 可以同时执行**的原理，实现**函数异步执行**

核心要点：充分利用**计算机 CPU 核心（单核 / 多核）**对**程序运行速度、架构设计**进行**组合优化**的方案选型。

> [!NOTE]
>
> 优化方式：
>
> - 使用 **Lock/RLock 互斥锁**对**“临界区” 资源加“锁”**，防止**冲突访问**
> - 使用 **Queue 队列**实现**不同进程/线程**之间的**数据通信**，实现 **生产者-消费者模式**
> - 使用**进程池 Pool** 和 **线程池 Pool**，**简化进程/线程**的任务提交、等待结束、获取结果...等操作

##### 程序的并发粒度

根据 CPU 核心（单核/多核）的执行机制，在并发编程中，程序的并发粒度如下：

- 多机器**并行**：例如大数据技术（Hadoop、Hive、Spark），可以将成千上万份数据**分布式**地放在**不同的机器**来**并行**计算
- 多进程**并行**：每个机器内，**启动多个 Python 进程提供服务（利用多核 CPU）**
- 多线程**并发**：**单个进程内，启动多个线程**提供服务
- 多协程**并发**：**单个线程内，使用协程机制**，提供**并发**能力

###### 并行与并发的核心区别

- **并行**：「多个执行单位」分别持有**各自独立的资源**
- **并发**：「多个执行单位」**共享同一空间内的资源**

<img src="C:/Users/win/Desktop/学习笔记体系/AI Agent/Python/03-Python Web 框架/FastAPI 异步框架/media/并发-并行的核心区别.png" style="zoom:67%;" />

#### 基本概念

- 程序的并发粒度有：**机器间、进程间、线程间、协程间**。

从**机器 > 进程 > 线程 > 协程**的顺序，程序运行花费的**资源调度耗费、程序上下文切换**的**耗费**，**从大到小递减**；

协程：

- 是在**一个线程内实现的并发**，协程的并发任务之间**调度**和**上下文的耗费最少**，对于**高并发**场景，可以认为**协程最高效率最快速**
- 核心要点：在于函数可以用 `async` 和 `await` 配合，在 **IO 阻塞等待期间交出线程执行权**，让**线程可以去处理其他用户请求**

所以，**协程减少了多机器/进程/线程的调度开销 + 切换不同上下文的开销**，更加**轻量级高性能**。

##### 多协程的 CPU 执行路径

核心要点：**CPU 执行路径只有一条**，但是**遇见 I/O 操作后便可以去执行其他的任务**，最终实现了**并发**。

## Web 开发概念

### Web 开发模式

Web 开发模式的发展经历了两大核心阶段：**前后端不分离**（传统的单体架构）与**前后端分离**（现代的主流架构）。

两者的核心区别在于**职责划分、技术栈解耦以及数据交互方式**。

#### 前后端不分离 (SSR - 服务端渲染)

在早期或传统开发中，前端（HTML/CSS/JS）和后端（Java/Python/PHP/Go等）混在同一个项目里。服务器不仅负责处理业务逻辑和数据库，还要**亲自拼接好 HTML 页面**，再整体返回给浏览器。

##### 工作角色

- **浏览器、App（前端）**
- **后端服务器**

##### **工作流程**

1. 浏览器向服务器发送 URL 请求（例如访问 `/index`）
2. 后端控制器（Controller）接收请求，调用数据库获取数据
3. 后端将数据注入到**HTML 模板引擎**（如 JSP, Thymeleaf, Django Template, Blade 等）中
4. 后端渲染出完整的 HTML 页面，一次性发送给浏览器
5. 浏览器直接解析并展示这个完整的 HTML

#### 前后端分离 (CSR - 客户端渲染)

现代开发中，前端和后端是**两个完全独立的项目**，甚至部署在不同的服务器上。前端负责“长相和交互”，后端负责“数据和逻辑”。它们之间唯一的纽带是 **API（通常是 RESTful API 或 GraphQL）**。

##### 工作角色

- **浏览器、App（前端）**：负责**发送 API 接口数据/静态资源请求**、填充并渲染 API 接口返回的数据
- **Web 服务器（`Nginx`）**：负责 **API 接口路由分发、静态资源部署**
- **后端服务器（Java/Python/PHP/Go 等语言编写的 Web 应用程序）**：负责**对外提供统一的 API 接口服务**，根据请求操作**返回所需响应数据（JSON、XML）**

**工作流程：**

1. 浏览器向**前端服务器**（如 Nginx 部署的 Vue/React 静态资源）请求页面
2. 前端服务器返回一个几乎空的 HTML 骨架和打包好的 JS 脚本
3. 浏览器解析 JS，发现需要数据，于是通过 Ajax/Axios 异步向**后端服务器**发起 API 请求
4. 后端服务器查询数据库，只返回纯粹的**数据（通常是 JSON 格式）**
5. 前端拿到 JSON 数据，利用 JS 动态把数据“塞进”页面中（客户端渲染）

### 通信架构模型

在现代（前后端分离）的 Web 通信架构模型中，通常抽象成以下 3 层架构模型：

- **客户端（Client）**：浏览器、手机 App、Postman
- **Web 服务器（Web Server）**：如 Nginx、Apache、Cloudflare
- **应用服务器与框架（Application Server & Framework）**：如 Uvicorn、Gunicorn，以及 FastAPI、Flask、Django

而将这 3 层抽象架构模型根据**底层的通信机制**拆分成如下 2 个**分层**部分：

- **客户端（Client）↔ Web 服务器（`Nginx`）**：采用 **`HTTP`、`WebSocket` 等协议**进行**通信**
- **Web 服务器（`Nginx`） ↔ 应用服务器与框架（如：FastAPI 框架程序提供的 API 接口）**：采用 **`WSGI`** 或 **`ASGI` 协议**进行**通信**

#### 客户端 App  ↔ Web服务器

Web 服务器通信协议是**客户端（如浏览器、App）**与 **Web 服务器（如 Nginx、Uvicorn）**之间进行数据交换的规则和约定。随着互联网的发展，这些协议经历了从简单到高效、从明文到加密的演进。

以下是 Web 服务器通信中核心的协议及其底层机制：

- **应用层协议**：
  - **`HTTP/1.1`、`HTTP/2`、`HTTP/3` 协议**
  - **`WebSocket`、`SSE` 双向通信协议**
- **`HTTPS` 安全层协议**
- **`TCP` & `UDP` 传输层协议**

##### 1. 核心应用层协议

###### HTTP 系列

HTTP（超文本传输协议）是 Web 通信的基石。

- **HTTP/1.1 —— 奠定基础**

  - **特点**：引入了 **持久连接（Keep-Alive）**，允许在同一个 TCP 连接上发送多个请求，避免了频繁的三次握手

  - **痛点**：**队头阻塞（Head-of-Line Blocking）**。在同一个 TCP 连接中，请求必须按顺序处理，如果前一个请求卡住，后面的请求就必须等待

- **HTTP/2 —— 性能飞跃**

  - **二进制分帧**：将数据拆分为更小的帧，并采用二进制编码，提高了解析效率

  - **多路复用（Multiplexing）**：允许在**同一个 TCP 连接上同时并发处理多个请求和响应**，彻底解决了 HTTP 层的队头阻塞问题

  - **头部压缩（HPACK）**：压缩 HTTP 头部，减少冗余数据传输

  - **服务器推送（Server Push）**：服务器可以主动向客户端推送资源

- **HTTP/3 —— 摆脱 TCP 束缚**

  - **基于 QUIC 协议**：摒弃了传统的 TCP，改用基于 **UDP** 的 QUIC 协议

  - **解决 TCP 队头阻塞**：在 TCP 中，如果一个数据包丢失，整个连接都会暂停等待重传。HTTP/3 的 QUIC 实现了连接内的多个流相互独立，某个流丢包不影响其他流

  - **极速建连**：将传输层握手和加密层（TLS）握手合并，实现 0-RTT 或 1-RTT 建立连接

###### 双向实时通信协议

传统的 HTTP 是“请求-响应”模式，只能由客户端发起。为了满足实时交互（如聊天、金融大盘、协同编辑）的需求，演进出了以下协议：

- **WebSocket**

  - **特点**：**全双工通信（Full-Duplex）**

  - **机制**：**基于 TCP**。客户端通过 HTTP 发起一个特殊的握手请求（Upgrade: websocket），握手成功后，连接升级为 WebSocket，**建立持久性的双向通道**，双方可随时发送数据

  - **优势**：协议头部非常小（仅几个字节），延迟极低，适合高频实时交互

- **SSE (Server-Sent Events)**

  - **特点**：**单向文本流（Server-to-Client）**

  - **机制**：**基于标准 HTTP 协议**，服务器使用 **`text/event-stream`** 格式持续向客户端推送数据

  - **优势**：实现简单，自带断线重连机制，适合不需要客户端高频回复的场景（如 AI 文本流式生成、新闻流推送）

##### 2. 安全层协议：HTTPS (TLS)

HTTPS 并非新协议，而是 **HTTP + SSL/TLS**。它在 HTTP 与 TCP 之间增加了一个加密层，确保数据的安全性。

核心安全机制

- **机密性（加密）**：混合加密。使用**非对称加密**（如 RSA、ECC）交换对称密钥，随后使用**对称加密**（如 AES）传输实际数据，兼顾安全与速度。
- **完整性（防篡改）**：通过 MAC（消息认证码）或哈希算法（如 SHA-256）确保数据在传输过程中未被篡改。
- **身份认证（防钓鱼）**：通过 **CA 数字证书** 验证服务器的真实身份。

##### 3. 传输层基石：TCP vs UDP

所有 Web 应用层协议最终都需要依托传输层来发送数据：

| **特性**          | **TCP (Transmission Control Protocol)** | **UDP (User Datagram Protocol)**   |
| ----------------- | --------------------------------------- | ---------------------------------- |
| **连接性**        | 面向连接（三次握手、四次挥手）          | 无连接                             |
| **可靠性**        | **绝对可靠**（重传机制、严格按序到达）  | 不可靠（尽力交付，可能丢包或乱序） |
| **速度**          | 较慢（有拥塞控制和流量控制机制）        | 极快（无握手，无控制机制）         |
| **对应 Web 协议** | HTTP/1.1、HTTP/2、WebSocket             | HTTP/3 (QUIC)                      |

##### 协议流转

$$
\text{应用层 (HTTP/2 / WebSocket)} \longrightarrow \text{安全层 (TLS 1.3)} \longrightarrow \text{传输层 (TCP)} \longrightarrow \text{网络层 (IP)}
$$

它们的核心区别在于**对异步（Async）和高并发能力的支持**。

##### API 接口

**应用程序编程接口（Application Programming Interface，API 接口）**：就是 Python 应用程序**对外提供**了一个**操作数据的入口**。

- 这个入口可以是**一个 `def` 函数或者 `class` 类方法，对应一个 URL 请求地址的映射关系**。（**URL 地址 → `def` 接口视图函数**）

当客户端向这个入口映射的 URL 地址**发送请求**时，Python 应用程序就会执行对应的 `def` 函数或者 `class` 类方法中的代码操作，完成对应的功能操作，并向客户端返回**响应数据**。

###### RESTful 接口规范

在**客户端**与**服务器端**之间进行 **API 接口的数据通信**时，需要符合一定的 **API 接口规范**。

- 目前现代主流的 API 接口实现规范主要有 **`RESTful`、`RPC`**

核心定义：**`RESTful`** 是**一种在 Web 开发中，定义 API 接口实现的设计规范**，尤其适用于**前后端分离（CSR）**的 Web 应用模式中，它是一种**面向资源**的编程方式。

核心理念：由**后端 Python 程序开发的 API 接口**是专门用来**提供数据资源操作**的，并**对外暴露一个数据资源的 API 访问接口（URL 地址）**；**客户端**则需要**访问这个 API 访问接口（URL 地址）**来表达**一个操作数据资源的请求**。

`RESTful` 接口规范：

| 请求方法     | 请求的 URL 地址（API 接口） | 描述                    |
| ------------ | --------------------------- | ----------------------- |
| **`GET`**    | `/xxx`                      | **获取、访问**-数据资源 |
| **`POST`**   | `/xxx`                      | **修改、新增**-数据资源 |
| **`PUT`**    | `/xxx`                      | **更新**-数据资源       |
| **`DELETE`** | `/xxx`                      | **删除**-数据资源       |
| ...          |                             |                         |

注：`RESTful` 是一种**通用**的接口规范，不限制编程语言与框架的使用。

##### WSGI 同步网关协议

**WSGI（Web Server Gateway Interface，Web 服务器网关接口）**：它是一个**介于 Web 服务器（`Nginx`、`uWSGI`） 与 Web 应用（`Flask`、`Django` 框架写的 Web 程序）之间**的一种**底层网关通信规范**。

核心概念：WSGI 是 Python 社区历史悠久的规范（PEP 3333），专为**同步（Blocking）请求**处理设计。

###### 工作机制

WSGI 采用经典的 **“一请求一线程”（Request-per-thread）** 或者是多进程模型

1. **服务器（如 Gunicorn、uWSGI）**接收到客户端的 HTTP 请求
2. 服务器调用 WSGI 应用程序提供的一个可调用对象（通常是一个函数或实现了 `__call__` 的实例）
3. 应用程序处理请求（此时线程会被数据库查询或 I/O 操作阻塞），处理完成后返回状态码和 HTTP 响应体

###### 核心局限

- **无法处理长连接**：由于采用阻塞模型，面对 **WebSocket**、**SSE（服务器发送事件）** 等需要长连接、双向通信的场景时，WSGI 会迅速耗尽服务器的线程池
- **高并发瓶颈**：当遇到密集的 I/O 等待时，线程只能死等，造成极大的资源浪费

###### 伪代码示例

```python
def simple_wsgi_app(environ, start_response):
    """标准的同步回调机制"""
    status = '200 OK'
    response_headers = [('Content-type', 'text/plain')]
    start_response(status, response_headers) # 同步调用服务器方法告知状态
    return [b"Hello World"] # 返回一个可迭代的字节序列
```

##### ASGI 异步网关协议

随着现代化 Web 需求（尤其是实时通信和 AI 流式渲染）的爆发，ASGI 诞生了。它是 WSGI 的精神继承者，旨在规范**异步** Python 通信。

###### 工作机制

ASGI 基于 Python 的 **`asyncio` 协程生态**，采用**单线程/少线程+事件循环（Event Loop）**的**异步非阻塞模型**。

1. **异步服务器（如 Uvicorn、Hypercorn）**接收请求
2. ASGI 应用接收一个包含连接信息的字典（`scope`）以及两个异步方法：`receive`（用于监听客户端发送的数据）和 `send`（用于向客户端发送数据）
3. 当遇到 **I/O 等待（如异步数据库查询）**时，当前协程会**主动挂起（yield）**，将控制权交还给事件循环（Event Loop）去处理其他请求

###### 核心优势

- **天生支持 WebSocket/SSE**：由于 `receive` 和 `send` 是持续生效的异步通道，ASGI 可以轻松维持成千上万个活跃的长连接
- **高并发能力**：在处理 I/O 密集型任务（如调用外部 AI 大模型 API、高频数据库读写）时，吞吐量远超 WSGI

###### 伪代码示例

```python
async def simple_asgi_app(scope, receive, send):
    """基于事件驱动的长连接/流式通道机制"""
    if scope['type'] == 'http':
        # 等待服务器准备好发送响应头
        await send({
            'type': 'http.response.start',
            'status': 200,
            'headers': [[b'content-type', b'text/plain']],
        })
        # 异步、分段地发送响应体（非常适合流式渲染，如大模型逐字打字输出）
        await send({
            'type': 'http.response.body',
            'body': b"Hello World",
        })
```

###### 完整数据流转

```python
[浏览器/客户端]
       │
       │  (1) 发起 WebSocket 握手 / HTTP 异步请求 
       │      采用：WebSocket / HTTP/2 协议 (网络通信协议)
       ▼
[ Nginx (反向代理) ]
       │
       │  (2) 转发请求给应用服务器 
       │      采用：TCP / Unix Socket (传输层)
       ▼
[ Uvicorn (ASGI 服务器) ]
       │
       │  (3) 解析网络协议，将其包装为标准格式
       │      调用：async def app(scope, receive, send)
       │      采用：ASGI 规范 (Python 异步接口标准)
       ▼
[ FastAPI (Python 框架) ] ── (4) 执行你的 async/await 业务逻辑
```

##### 核心对比

| **特性**       | **WSGI**                                          | **ASGI**                                       |
| -------------- | ------------------------------------------------- | ---------------------------------------------- |
| **设计核心**   | 同步、阻塞 I/O                                    | 异步、非阻塞 I/O                               |
| **标准提案**   | PEP 3333                                          | 独立于 PEP 之外的社区规范                      |
| **协议支持**   | 仅限 HTTP/1.1                                     | HTTP/1.1、**HTTP/2**、**WebSocket**            |
| **核心组件**   | 可调用函数 `application(environ, start_response)` | 异步函数 `async def app(scope, receive, send)` |
| **代表服务器** | Gunicorn, uWSGI, Werkzeug                         | Uvicorn, Hypercorn, Daphne                     |
| **代表框架**   | **Flask**, **Django** (传统版)                    | **FastAPI**, **Sanic**, **Django Channels**    |

#### 网络通信协议与网关协议的组合

不同的网络通信协议，其特点不同（同步或异步、单向或双向），这也直接决定了它们必须匹配对应的网关接口：

##### 传统 HTTP/1.1  + WSGI

- **场景**：传统的网页渲染、CRUD 接口
- **关系**：HTTP/1.1 的请求-响应模型是典型的“一问一答”同步模式。当 Nginx 收到一个 HTTP 请求后，**WSGI 服务器**（如 Gunicorn）将其解析为 Python 的 `environ` 字典，同步传给 Flask/Django 框架，处理完后原路返回
- **局限**：如果客户端用 HTTP/1.1 维持了一个长连接（Keep-Alive），或者想做 WebSocket 双向通信，WSGI 的同步阻塞模式会直接让一个线程卡死在那里，无法应对

##### HTTP/2 / HTTP/3 / WebSocket + ASGI

- **场景**：大模型 AI 文本流式输出（SSE）、实时聊天、大文件分片上传
- **关系**：这些现代协议的核心特点是**异步、双向、多路复用**。**ASGI 服务器**（如 Uvicorn）能够理解这些复杂的协议细节，它把一个 WebSocket 连接或 HTTP/2 的多个流，抽象为 ASGI 规范中的 `scope`、`receive` 和 `send` 异步通道
- **优势**：当服务器通过 HTTP/3 收到大模型返回的流式文本数据时，ASGI 框架（如 FastAPI）可以用 `await send()` 逐字推给客户端，期间不阻塞任何线程，把现代通信协议的高并发优势发挥到极致

#### 与 Starlette 的关系

核心定义：**Uvicorn 是“服务器”，而 Starlette 是“Web 框架”**。它们两个组合在一起，构成了现代 Python 异步 Web 开发（比如 FastAPI）的核心基石。

- 共同点：Uvicorn 和 Starlette 两者都是基于 **ASGI 标准协议** 实现的。

##### 什么是 ASGI 标准？

**ASGI （Asynchronous Server Gateway Interface，异步服务器网关接口）**是 Python 社区为了让 **Web 服务器**和**异步 Web 应用程序**之间能够**互相通信**而制定的一套**标准协议**。

- **Uvicorn 是 ASGI 的“实现者”（服务器）**
- **Starlette 是 ASGI 的“应用者”（框架）**

因为两者都遵守同一种语言（ASGI 协议），所以 **Uvicorn 可以完美地驱动 Starlette**。

###### 为什么选择 ASGI 而不是 WSGI？

传统的 **WSGI** 标准是**同步**的，处理一个请求必须等待响应完全返回，这导致它难以支持 WebSockets 或 HTTP 长轮询。

而 **ASGI** 是**异步**的，它让 Python Web 应用能像 Node.js 或 Go 一样，在处理等待数据库或网络 I/O 时，同时处理其他请求，大大提高了并发能力。

##### 职责分工

###### Uvicorn-专注性能的底层服务器

Uvicorn 是一个超高性能的 **ASGI HTTP 服务器**。它的核心任务非常纯粹：

- **监听端口**：比如在本地监听 `127.0.0.1:8000`
- **处理网络原始数据**：把网络上传过来的 TCP 字节流解析成 HTTP 请求（或者 WebSocket 数据包）
- **转化为标准格式**：把这些请求打包成 ASGI 协议规定的 Python 字典（即 `scope`, `receive`, `send` 协议三要素）
- **扔给上层框架**：它不关心 API 路由长什么样，它只负责把打包好的 ASGI 数据塞给像 Starlette 这样的框架

###### Starlette-专注业务的轻量级 Web 工具箱

Starlette 是一个轻量级、高性能的 **ASGI 框架/工具包**。当它从 Uvicorn 那里接过 ASGI 数据后，开始干脏活累活：

- **路由分发 (Routing)**：检查请求的路径（比如 `/items/1`），决定由哪个 Python 函数来处理
- **生命周期管理 (Lifecycle)**：处理应用的启动（Startup）和关闭（Shutdown）事件（比如连接和断开数据库）
- **高级抽象**：把 Uvicorn 传过来的原始数据封装成更好用的 `Request` 和 `Response` 对象
- **常用功能**：自带 Session 支持、Cookie 处理、异常处理、CORS 跨域配置以及 WebSockets 的高级封装

###### 与 FastAPI 的 “套娃关系”

- **`FastAPI` = `Starlette`（Web 核心）+ `Pydantic`（数据校验）+ 自动生成 OpenAI 文档等高级特性**

当运行一个 FastAPI  项目时，底层的调用链如下：

```python
+-------------------------------------------------------------+
| 1. 用户浏览器发起请求 (HTTP / WebSocket)                     |
+-------------------------------------------------------------+
                              │
                              ▼
+-------------------------------------------------------------+
| 2. Uvicorn (服务器)                                         |
|    - 监听端口，将原始网络字节流解析为 ASGI 格式                |
+-------------------------------------------------------------+
                              │
                              ▼
+-------------------------------------------------------------+
| 3. Starlette (FastAPI 基类)                                 |
|    - 进行路由匹配、处理中间件、管理异常                     |
+-------------------------------------------------------------+
                              │
                              ▼
+-------------------------------------------------------------+
| 4. FastAPI (上层封装)                                       |
|    - Pydantic 校验入参 -> 执行你的业务函数 -> 格式化输出数据  |
+-------------------------------------------------------------+
```

核心结论：

- **Uvicorn** 负责 **“怎么传”**（网络传输、高性能异步 IO）
- **Starlette** 负责 **“怎么管”**（路由、请求、逻辑控制）

它们之间通过 **ASGI 协议** 紧密相连。

## 工程化选型

FastAPI 团队提供了 `fastapi` 和 `fastapi[standard]` 两个标准库，两者在框架作用、代码上完全相同，只不过实际应用场景不同。

### `fastapi` vs `fastapi[standard]`

定义：`fastapi[standard]` 和 `fastapi` 的核心代码**完全相同**，它们唯一的区别在于**默认安装的外围依赖包（Extras）不同**。

- 这是 FastAPI 在较新版本中引入的变化，目的是让开发者在“快速开发”和“轻量部署”之间有更灵活的选择。

| **特性**            | **pip install fastapi**【推荐】                              | **pip install "fastapi[standard]"**【不推荐】                |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **定位**            | **轻量版 / 生产环境精简版**                                  | **标准版 / 完整开发版**                                      |
| **包含的内容**      | 仅包含 FastAPI 核心框架及最基础的依赖（如 Pydantic、Starlette） | 包含 FastAPI 核心，以及开发和运行所需的**一整套常用工具链**  |
| **内置 Web 服务器** | ❌ **不包含** `uvicorn`。你无法直接运行 `fastapi dev` 或 `uvicorn` 命令，需要手动单独安装服务器 | **包含** `uvicorn`。安装后可以直接使用 `fastapi dev` 启动项目 |
| **适用场景**        | 适合部署到 Docker 容器、生产环境，或者希望完全自主控制依赖版本的场景 | 适合本地开发、新手入门、快速原型构建                         |

### `fastapi[standard]` 库（本地）

*仅用于本地项目轻量级开发，不适合上线生产项目*

#### 核心定义

`FastAPI` 本身是一个库，所以需要通过 `pip` 来安装、下载到本地项目中进行集成：

```bash
$pip install 'fastapi[standard]'
```

当选择安装 `fastapi[standard]` 时，Python 会**自动捆绑安装**以下常用库（免去了手动一个个 `pip install` 的麻烦）：

- **`uvicorn`**：高性能的 ASGI 服务器，用于运行 FastAPI 应用
- **`fastapi-cli`**：**命令行工具**；能直接在终端使用 **`fastapi dev main.py`** 或 **`fastapi run main.py`**
- **`httpx`**：用于测试 API 接口（`TestClient` 会用到）
- **`jinja2`**：如果需要渲染 HTML 模版
- **`python-multipart`**：如果需要处理表单提交或文件上传

#### `fastapi-cli` 脚手架工具

> [!NOTE]
>
> 在 `fastapi-cli` 工具出现之前，社区里有各种各样的第三方脚手架（如 `fastapi-boilerplate`），或者大家各自编写复杂的 `Makefile` / `shell` 脚本来启动项目。官方推出 `fastapi-cli` 的主要原因有两点：
>
> 1. **统一开发者体验（DX）**：像 Node.js 的 `npm run dev` 或 Go 的 `go run` 一样，Python 开发者现在也有了官方统一的、极简的启动命令
> 2. **平滑的生产环境过渡**：新人在本地用 `fastapi dev` 顺手了，到服务器上只需要无脑换成 `fastapi run`，不需要去心智构建“本地用 uvicorn，生产用 gunicorn”的复杂逻辑

核心定义：在 FastAPI 较新的版本中，官方推出了 **`fastapi-cli`**。它不再只是一个单纯的“启动命令”，而是演变成了 FastAPI 官方自带的**项目脚手架与生命周期管理工具**。

核心价值：在过去，启动 FastAPI 必须依赖 **`uvicorn main:app --reload`** 这种冗长的命令，而现在，**`fastapi-cli`** 将这些操作**标准化**了。

##### 核心功能及命令

当安装了 `fastapi[standard]`，或者单独安装了 **`pip install fastapi-cli`** 后，就可以在终端直接使用 **`fastapi` 命令**了。

它主要提供了两个最核心的子命令：

###### fastapi run 生产启动

用于在**生产环境**中启动 FastAPI 应用。它会自动检测生产环境的最佳配置（例如默认关闭代码热重载，优化日志输出）。

```bash
$fastapi run <.py 模块名>
```

**特点**：默认在 `0.0.0.0:8000` 端口启动，适合直接在 Docker 容器或云服务器中运行。

###### fastapi dev 本地启动

专门为**本地环境**的开发者优化、调试，提供了非常友好的交互体验。

```bash
$fastapi dev <.py 模块名>
```

作用：

- **自动热重载（Auto-reload）**：当修改并保存代码时，服务会自动重启

- **漂亮的控制台日志**：相比以前 Uvicorn 简陋的日志，`fastapi dev` 提供了带颜色高亮、格式更清晰的请求日志，甚至能直接看出请求的路由和状态码

- **默认本地端口**：默认在 `127.0.0.1:8000` 启动

##### 局限性

虽然它很方便，但在**真实的企业级生产环境中，许多团队依然持保留态度**。

- ✅ 推荐使用 `fastapi-cli` 的场景：

  - **新项目的快速原型（POC）**：写 Demo 或个人项目时，效率极高

  - **轻量级 Docker 部署**：在 Dockerfile 的最后一行，直接写 `CMD ["fastapi", "run", "main.py"]` 非常清爽，它会自动绑定程序运行地址设为 `0.0.0.0`

- ❌ 企业级生产环境的局限性：

  - **黑盒化严重**：`fastapi run` 的底层依然是调用 Uvicorn 这个 Web 框架，但它把 Uvicorn 的很多底层参数（如 `--workers`、`--limit-concurrency` 等）给“包”起来了。对于需要精细化调优高并发参数的运维和架构师来说，这种封装反而不方便

  - **无法替代专业的项目脚手架**：需要注意的是，`fastapi-cli` 目前**只是一个“启动和运行”工具，而不是一个“项目结构生成器”**。它不会像 Django 的 `django-admin startproject` 那样自动生成包含 `models/`, `views/`, `config/` 的企业级标准目录结构

综上所述，**`fastapi-cli` 仅适合用于本地项目的开发，不适合用于生产项目**。
