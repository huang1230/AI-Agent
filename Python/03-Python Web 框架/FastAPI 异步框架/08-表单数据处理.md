# 表单数据

## 前置概念

### `Content-Type` 传输格式

在 **HTTP 协议通信**中，**服务器端**会**根据客户端发送的 HTTP 请求头中的 `Content-Type` 字段（数据编码）**来**解析对应的二进制数据包**：

- **普通 Body 请求体**：**`Content-Type: application/json`**

  - **数据格式**：采用 **`{ key:value }` JSON 格式**的字符串传输数据

  - 抓包示例：

    ```ini
    POST /api/user HTTP/1.1
    Host: example.com
    Content-Type: application/json
    
    {
      "name": "Alex",
      "age": 25,
      "skills": ["Python", "Go"]
    }
    ```

- **带有表单数据的 Body 请求体**：

  - **`Content-Type: application/x-www-form-urlencoded`** **（普通表单数据）**

    - **数据格式**： 采用 **`key=value` URL 编码格式**传输数据；非 ASCII 字符（如中文、特殊符号）会被进行 URL 编码（%XX）

    - 抓包示例：

      ```ini
      POST /api/login HTTP/1.1
      Host: example.com
      Content-Type: application/x-www-form-urlencoded
      
      username=Alex&password=123456&remberMe=true
      ```

  - **`Content-Type: multipart/form-data`**：**（分块流式传输-表单数据）**

    适用于带有**文件上传**的表单数据，数据包内同时包含**『普通文本』**和**『文件二进制流』**数据。

    - **数据格式**：根据提交的**表单字段（Form Field）个数**采用**多个随机的隔离字符（`boundary`）**进行**分块流式传输**

    - 抓包示例：

      ```ini
      POST /api/upload HTTP/1.1
      Host: example.com
      Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW
      
      ------WebKitFormBoundary7MA4YWxkTrZu0gW
      Content-Disposition: form-data; name="username"
      
      Alex (这里是表单中的普通文本字段数据...)
      ------WebKitFormBoundary7MA4YWxkTrZu0gW
      Content-Disposition: form-data; name="avatar"; filename="me.png"
      Content-Type: image/png
      
      (这里是图片的二进制数据...)
      ------WebKitFormBoundary7MA4YWxkTrZu0gW--
      ```

| **Content-Type**                        | **适用场景**                     | **传输效率与特点**                                           |
| --------------------------------------- | -------------------------------- | ------------------------------------------------------------ |
| **`application/json`**                  | 现代前后端分离项目、API 接口交互 | 支持**复杂嵌套**，可读性最好，前后端解析成本低               |
| **`application/x-www-form-urlencoded`** | 简单的键值对、登录表单           | **结构扁平**，无法优雅处理深层嵌套，传输大文件时**效率极低** |
| **`multipart/form-data`**               | 文件、图片、视频上传，或图文混传 | 通过 **`boundary` 分块传输**，支持二进制流，用于**上传文件** |

### python-multipart 依赖

在 FastAPI 中，如果想接收前端传来的**表单数据（Form Data）**或 **文件（File）**，就必须安装 **`python-multipart`** 这个第三方依赖。

```bash
$pip install python-multipart
```

核心作用：**`python-multipart` 是一个底层的 Python 解析器，专门用来把浏览器发送的复杂表单编码数据，翻译成 Python 能够理解的字典、字符串或文件对象。**

- **`python-multipart`** 是一个**底层依赖**，只需安装即可，**无需显式引入使用**，它内部会**自动运行**。

> 如果不安装 `python-multipart`；当在 FastAPI 中使用 `Form()` 或 `File()` 时，程序就会直接报错：
>
> `RuntimeError: Form data requires "python-multipart" to be installed.`

#### 为什么需要它？

在前端使用 Form 表单格式发送数据时，浏览器并不会发送 JSON，而是会把数据打包成以下两种格式之一：

- **`application/x-www-form-urlencoded`（纯文本表单）**

  这是最常见的**普通表单（不带文件）**。浏览器会将数据拼接成类似 URL 参数的字符串。

  - **浏览器发送的内容**：`username=admin&password=secret123&age=25`
  - **`python-multipart` 的作用**：在后台把这串密密麻麻的字符串切割、解码，**转化为** FastAPI 可以直接使用的 **Python 变量**

- **`multipart/form-data`（带文件的混合表单）**

  当需要**上传图片、文档**等文件，或者同时**上传“文本+文件”**时，浏览器必须使用这种格式。

  - **浏览器发送的内容**：由于文件是二进制数据，浏览器会使用一个随机生成的“分界线（Boundary）”把不同的字段隔开，数据看起来像一堵杂乱的文本和二进制墙
  - **`python-multipart` 的作用**：像手术刀一样，精准地按照分界线把“文本字段”和“二进制文件流”剥离出来，让 FastAPI 能够**把文件封装成方便操作的 `UploadFile` 对象**

#### 数据的流转过程

```python
[ 浏览器 / 前端 ]
       │
       │ (提交表单: application/x-www-form-urlencoded 或 multipart/form-data)
       ▼
[ FastAPI / Starlette 框架 ]
       │
       │ 🔍 发现请求体是表单格式 -> 自动调用底层依赖：
       ▼
┌──────────────────────────────────────┐
│        python-multipart              │
│    🛠️ 解析原始请求体，将其拆解为键值对    │
└──────────────────────────────────────┘
       │
       │ (解析完毕，转换为干净的 Python 数据)
       ▼
[ API 路由函数 ] -> 拿到 `username: str` 和 `avatar: UploadFile`
```

## 普通纯文本表单（`k=v&...`）

在 FastAPI 中，要想声明接收一个或多个表单数据字段以及参数值校验主要通过 **`fastapi.Form` 类** 以及 `Pydantic` 数据模型实现。

- **Form Data 表单数据**：是**一个 `key=value&key2=value2...` 纯文本的键值对集合，位于 Request Body 请求体部分**

### `Form` 类（包装、校验）

FastAPI 支持通过在 API 接口函数中的**「形参列表」**中**定义一个或多个 `形参变量`**来**表示 Request Body 请求体参数（数据字段）**。

- **`Form()`** 可以实现对 **Request Body 请求体的某个参数值（`FormData` 表单数据的一个数据字段）**进行**包装声明**及**值校验处理**。 

关键点：`Form` 是**继承自 `Body` 的子类**，表示为**表单数据**是存储于 **Request Body 请求体中的一种形式**。

#### 基本语法

```python
from fastapi import FastAPI, Form

<FastAPI 应用实例> = FastAPI()

@<FastAPI 应用实例>.post('<URL 路径>')
async def <API 路由函数>(<形参变量x>: <数据类型> = Form()): 
    # 不加 Form() 是 Query 查询参数，加了 Form() 则是请求体中的一个表单数据字段

    return <响应数据>
```

核心机制：**`<形参变量x> = Form()`** 的主要目的是告诉 FastAPI 框架：“这个 **<形参变量x> 的值**应该**从 HTTP 请求中的 Request Body 请求体部分（Payload）**中以**表单数据（`k=v`）**的格式进行**解析提取**。”

> 注：如果**不使用 `Form()`**，FastAPI 会**默认**把 `<形参变量x>` 当作一个**Query 查询参数（Query Parameter）**。

##### 命名匹配机制

```ini
<形参变量x>: <数据类型> = Form()
<形参变量x>: Annotated[<数据类型>, Form()]
```

FastAPI 规定，在 **API 接口函数**中定义的 **`<形参变量x>`** 必须**与 HTTP 请求中 Request Body 请求体的 FormData 表单数据字段同名**：

- **✅️同名**：FastAPI **正确匹配到 Body 请求体中对应的 FormData 表单数据字段名，解析注入**
- **❌️不同名**：FastAPI **无法匹配到对应的  Body 请求体中对应的 FormData 表单数据字段名**，抛出 **`422 Field required` 异常**

```python
[ 客户端：HTTP Form 请求体 ] ───> 携带表单数据 user_name=apple&...
                                                  │
                                          (寻找同名字段)
                                                  │
                                                  ▼
[ FastAPI 接口函数 ] 
  ├── ❌ 失败案例: def read_items(name: str = Form())
  │               └── 结果: 找不到表单里的 'name' ──> 💥 抛出 422 Field required 异常
  │
  └── ✅ 成功案例: def read_items(user_name: str = Form())
                  └── 结果: 成功匹配 'user_name' ──> 🎉 形参 user_name 成功注入 'apple'
```

解决方案：可以使用 **`Form()` 类的 `alias` 参数设定别名**，让 **`alias` 别名去匹配 FormData 表单数据字段名**，而 **API 接口函数中的 `形参变量` 则使用其他名称**。

```python
[ 客户端：HTTP Form 请求体 ] ───> 携带表单数据 user_name=apple&...
                                                  │
                                          (改去匹配 alias 的值)
                                                  │
                                                  ▼
[ FastAPI 接口函数 ]
  └── def read_items(name: Annotated[str, Form(alias="user_name")])
                      │                     └─ 1. 发现配置了别名 "user_name"
                      │
                      └─ 2. ✅ 别名成功匹配表单字段，将 "apple" 注入给内部变量 name
```

##### 自动类型转换

- **自动类型转换**：FastAPI 会**根据类型注解（Type Hints）自动转换 Form 表单数据字段的数据类型**，**失败则返回 422 异常**。

```python
@app.post("/login")
async def login(id: int = Form(...), username: str = Form(...), password: str = Form(...)):
    """
    登录接口
    :param username: 用户名

    :param password: 密码

    :return:
    """
    return {"username": username, "password": password}
```

规则：

- 若传入 `username=xxx&password=123123&id='123'` ，FastAPI 会将其 `'123'` **自动处理为数值**

  ```ini
  POST /login HTTP/1.1
  Content-Length: 35
  Content-Type: application/x-www-form-urlencoded
  Host: 127.0.0.1:8000
  username=xxx&password=123123&id='123' # 这里的 id 会被自动转为 int 类型的值 123
  ```

- 若**类型不匹配（如 `'id'` 字段为`int` 类型，但传入非数字）**，则**响应体**会返回 **422 异常**

  ```ini
  POST /login HTTP/1.1
  Content-Length: 35
  Content-Type: application/x-www-form-urlencoded
  Host: 127.0.0.1:8000
  username=xxx&password=123123&id=qwe # 这里 id 是错误的值，qwe 不能被转为 int 类型
  ```

```assembly
Status Code: 422
Error: Unprocessable Content

Response body
{
  "detail": [
    {
      "type": "int_parsing", # 异常类型
      "loc": [
        "body", # 异常位置
        "id" # 异常字段
      ],
      "msg": "Input should be a valid integer, unable to parse string as an integer", # 异常信息
      "input": "qwe" # 异常输入内容
    }
  ]
}
```

#### 底层工作机制

> “**表单输入 ➔ 转换为 Python 变量 ➔ JSON 输出**”

1. **初始化启动，`Form()` 包装声明**

   当启动 FastAPI 应用时，FastAPI 会**根据 `Form()` 类识别**表示**当前 API 接口要接收的 HTTP 请求体是一个 `Content-Type: application/x-www-form-urlencoded`  纯文本 URL 编码格式（`k=v`）的表单数据**。

2. **HTTP 请求解析**

   - 当**收到客户端发来的一个 HTTP 请求**时，FastAPI **识别检测**到携带的 **`Content-Type` 请求头字段**是 **`multipart/form-data`** 

   - FastAPI 会**借助 `python-multipart` **第三方底层依赖，**对 Request Body 请求体中**的**原始二进制表单数据**进行**解析**：

     ```ini
     POST /register4 HTTP/1.1
     Content-Length: 40
     Content-Type: application/x-www-form-urlencoded
     Host: 127.0.0.1:8000
     username=jack&password=12345678&code=103
     ```

   - 根据 **`k=v`** 的形式**解析**，并**转换包装**为一个 **Python 字典（`{k:v}`）**。

3. **`Form()` 匹配并解析注入**

   FastAPI 会**使用 `Form()` **，在转换后的 **Python 字典中依次匹配与 `形参变量` 同名的表单数据字段名**，并**将匹配到的表单字段值注入**到 API 接口函数中的 **`<形参变量>` 中**。

4. **API 接口函数返回 JSON 格式的响应数据包**

   在 API 接口处理完逻辑返回响应后，**响应数据包**中的**`Content-Type` 响应头字段依然还是 `application/json`**，目的是为了**方便客户端以 JSON  的格式解析响应体**。

**设计哲学**：**“请求格式与响应格式完全解耦”**。

**核心机制（输入不等于输出）**：

​	客户端可以使用**表单（Form）**、**文件（File）** 甚至 **XML（自定义解析）** 各种奇形怪状的格式把数据送进来；但在经过 FastAPI 的统一洗礼后，进到你业务代码里的都是标准 Python 变量，而返回给前端的又是标准统一的 **JSON**。这种设计极大地降低了前后端的对接成本。

#### 包装、校验

核心要点：`Form()` 是一个集**位置声明（包装）、安全把关（参数值校验）**和**代码即文档（元数据描述）**于一体的生产力工具。

核心定义：在 FastAPI 中，**`Form()` 函数用于显式声明一个形参应该从 HTTP 请求体（Body）的表单数据中提取解析**。它除了继承自 Pydantic 的基础校验参数（如 `default`, `description`）之外，还包含了一些专门用于控制**请求体解析行为**和 **OpenAPI/Swagger 文档生成**的高级参数。

`Form` 类提供了如下校验方式：

```python
Form(
	default: Any = Undefined, # 静态默认值
    default_factory: (() -> Any) | None = _Unset, # 动态默认值（工厂）
    media_type: str = 'application/x-www-form-urlencode', # 预期接收的数据格式（Content-Type）
    
    # 长度限制，适用于 str、list\dict\set\tuple 等序列容器数据类型
    min_length: int | None = None, # 最小长度
    max_length: int | None = None, # 最大长度
    
    # 数值范围验证，适用于 int、float 数值类型
	gt: float | None = None, # 大于 ＞
    ge: float | None = None, # 大于等于 ≥
    lt: float | None = None, # 小于 ＜
    le: float | None = None, # 小于等于 ≤
    
    # 正则表达式校验
    pattern: str | None = None,
    regex: str | None = None,
    
    # 其他（Swagger UI 文档标注）
    alias: str | None = None, # 别名
    title: str | None = None, # 参数标题
    description: str | None = None, # 参数描述信息
    deprecated: bool | None = None, # 是否已被废弃
    #...
)
```

### 声明接收方式

FastAPI 中，声明接收 Form 表单字段的写法有 2 种，就像 `Path()\Query()\Body` 一样：

#### Form() 散装式包装读取

直接在 API 接口函数中，写**多个 `<形参变量>: <数据类型> = Form()`** 来**逐个接收读取并解析注入**，适用于**少量 Form 表单字段** 的操作。

```ini
# HTTP 请求
POST /register HTTP/1.1
Content-Length: 79
Content-Type: application/x-www-form-urlencoded
Host: 127.0.0.1:8000
username=jack&password=12345678&email=1046%40qq.com&mobile=17673108812&code=103
```

- ❌️**传统写法 `<形参变量>: <数据类型 | None> = Form(None)`**：

  ```python
  @app.post('/register')
  async def register(
      username: str = Form(),
      password: str = Form(..., min_length=8),
      email: str = Form(),
      mobile: str = Form(..., min_length=11, max_length=11),
      code: int | None = Form(None)  # 非必填
  ):
      return {"msg": "注册成功"}
  ```

- ✅️**现代写法 `<形参变量>: Annotated[<数据类型 | None>, Form()] = <默认值 | None>`**：

  ```python
  @app.post('/register2')
  async def register2(
      username: Annotated[str, Form()],
      password: Annotated[str, Form(..., min_length=8)],
      email: Annotated[str, Form()],
      mobile: Annotated[str, Form(..., min_length=11, max_length=11)],
      code: Annotated[int | None, Form()] = None  # 非必填
  ):
      return {"msg": "注册成功"}
  ```

#### Pydantic 模型声明

*适用于多个 Form 表单字段 操作*

核心要点： 默认情况下在 API 接口函数中定义的一个 Pydantic 模型会被**默认解析为 JSON**，需要额外使用  **`Form()`** 或 **`Depends()`** 进行 **一次性包装声明** 。

FastAPI 会**自动解析**这个由 `BaseModel` 创建的**Pydantic 数据模型类结构**，并**从 HTTP 请求中批量读取 Request Body 请求体的表单数据字段并依次解析注入到 API 接口的 `形参变量` 中** 。

###### 模型定义

> ⚠️注意点：**HTTP 表单（`application/x-www-form-urlencoded`）**本身是**扁平的键值对结构**，**不支持原生嵌套对象**；
>
> 如非必要，要**想传入嵌套结构**，需转为 **`application/json` JSON 格式传输**。

```python
from fastapi import FastAPI

<FastAPI 应用实例> = FastAPI()

# 数据模型A
class <数据模型A>(BaseModel):
    # 模型字段...
    <字段名>: <数据类型> = Field()
    <字段名>: <数据类型> = Field()
    # ...
```

HTTP 请求：

```ini
POST /register3 HTTP/1.1
Content-Length: 79
Content-Type: application/x-www-form-urlencoded
Host: 127.0.0.1:8000
username=jack&password=12345678&email=1046%40qq.com&mobile=17673108812&code=103
```

模型定义：

```python
class RegisterForm(BaseModel):
    username: str = Field(title="用户名")
    password: str = Field(..., title="密码", min_length=8)
    email: str = Field(title="邮箱")
    mobile: str = Field(..., title="手机号", min_length=11, max_length=11)
    code: int | None = Field(None, title="验证码")  # 非必填
```

##### Form() 批量包装声明

- ❌️**传统写法 `<形参变量>: <数据模型> = Form()`**：

  ```python
  @app.post('/register3')
  async def register3(form: RegisterForm = Form(description='注册表单')):
      print(form.model_dump())
      # {'username': 'jack', 'password': '12345678', 'email': '1046@qq.com', 'mobile': '17673108812', 'code': 103}
  
      return {"msg": "注册成功"}
  ```

- ✅️**现代写法 `<形参变量>: Annotated[<数据模型>, Form()] = <默认值>`**：

  ```python
  @app.post('/register4')
  async def register4(form: Annotated[RegisterForm, Form(description='注册表单')]):
      print(form.model_dump())
      # {'username': 'jack', 'password': '12345678', 'email': '1046@qq.com', 'mobile': '17673108812', 'code': 103}
  
      return {"msg": "注册成功"}
  ```

##### Depends() 批量包装声明

*这是一个 FastAPI 历史遗留的黑魔法写法*

###### 基本概念

​	**早期的 `Form()`** 只能用来**修饰**并解析包装**单个表单字段**的变量，且**不支持解析 Pydantic 数据模型**（默认会被**解析成 JSON 格式**）；

​	于是，那时候开发者们使用的是 **`Depends()`** （黑魔法）以**依赖注入**的形式**逐个解析 Pydantic 数据模型**中的 数据字段，再**重组平铺**注入**依次添加 `Form()` 单个声明**，以此作为 API 接口函数中的 Form 表单字段；

​	但是这会造成一个问题：**`Depends()` 破坏了代码的语义**——明明是“表单字段（Form）”，代码里写的却是“依赖（Depends）”‘；

​	直到 2024 年底，FastAPI 官方重构了底层架构，**允许 `Form()`** 也**修饰并解析包装一整个 Pydantic 数据模型的 JSON 结构体**。

简单来说，`Form()` 是现代写法（*推荐*），`Depends()` 是老旧的 Hack 写法。

###### 示例

- **模型定义**：

  > 需要额外定义一个**类方法**，用于 **`Depends()` 调用解析并重组**

  ```python
  class RegisterForm(BaseModel):
      username: str
      password: str
      email: str
      mobile: str
      code: int | None
  
      @classmethod
      def as_form(
          cls,
          username: str = Form(title="用户名"),
          password: str = Form(..., title="密码", min_length=8),
          email: str = Form(title="邮箱"),
          mobile: str = Form(..., title="手机号", min_length=11, max_length=11),
          code: int | None = Form(None, title="验证码")  # 非必填
      ):
          return cls(
              username=username,
              password=password,
              email=email,
              mobile=mobile,
              code=code,
          )
  ```

- **✅️通用稳定写法 `<形参变量>: Annotated[<数据模型>, Depends(<数据模型类>.<解析类函数>)]` 包装声明注入**：

  ```python
  @app.post('/register5')
  # 或者 form: RegisterForm = Depends(RegisterForm.as_form)
  async def register5(form: Annotated[RegisterForm, Depends(RegisterForm.as_form)]):
      print(form.model_dump())
      # {'username': 'jack', 'password': '12345678', 'email': '1046@qq.com', 'mobile': '17673108812', 'code': 103}
  
      return {"msg": "注册成功"}
  ```

  `Depends(RegisterForm.as_form)` 模式：

  1. **`Form(...)` 负责拦截**：利用 `Depends` 依赖注入，强行让 FastAPI 在底层把表单里的**每一个 key/value 解构出来**
  2. **`cls(...)` 负责打包**：把解构出来的零散变量，再**重新塞进 `RegisterForm` 模型**中，最后享受 Pydantic 强大的数据校验

  虽然写起来麻烦，但它在老版本的 FastAPI 中**极其稳定**，能够完美兼顾“表单接收”和“模型校验”。

##### Form() vs Depends()

| **维度**         | **老写法 (Depends + <解析类函数>)**              | **新写法 (Annotated[Model, Form()])**     |
| ---------------- | ------------------------------------------------ | ----------------------------------------- |
| **代码量**       | 臃肿、需要手写解构映射                           | 极其精简，符合 Python 直觉                |
| **版本兼容性**   | **无敌。** 哪怕是几年前的古董 FastAPI 版本也能跑 | 必须要求**较新的 FastAPI 版本**           |
| **API 文档生成** | 完美生成 Swagger 文档中的每一个表单输入框        | 完美生成 Swagger 文档中的每一个表单输入框 |
| **复杂嵌套支持** | 支持在类方法里写自定义的**复杂转换逻辑**         | 只能处理**扁平**的表单字段                |

- 只要 FastAPI 版本不是特别古老，**果断抛弃老写法（`Depends()`），拥抱新写法（`Annotated[RegisterForm, Form()]`）！**

## 混合表单（带文件上传）

### 前置概念

在 FastAPI 中，主要有以下 2 种方式来声明接收上传的文件：

- **`File()`**：
  - 一个**类函数**；用于告诉 FastAPI **应该从 HTTP 请求的 `multipart/form-data` 表单请求体中去提取原始二进制流数据**
  - `File()` 类**继承**了 FastAPI 底层的 **`FieldInfo` 元信息基类**，可以传入一些**值校验、元信息描述等参数配置**，可用于 Swagger 文档渲染
  - 使用 **`bytes` 类型**来定义**接收原始二进制流的文件数据**
  - 用于**极小文件上传**，会**一次性将二进制流数据存入内存中**
- **`UploadFile`**：
  - 一种**数据类型**；
  - 决定了 FastAPI 把**收到的原始二进制流的文件数据**进行**解析包装成一个 Python 标准的文件对象**
  - 额外还**提供**了类似 Python 标准文件对象的 API（`filename` 文件名称、`size` 文件大小...）
  - 用于**大文件上传**，采用**分块传输二进制流**机制，一旦**超过分块传输的长度限制**则会**先写入硬盘的临时内存**中，**防止内存溢出**

注：通常更推荐使用 **`UploadFile`**，因为它有很多优势，比如文件不会全部加载到内存中（对大文件很友好），并且提供了类似 Python 标准文件对象的 API。

#### 注意点

在同一个路由函数里，**不能**把 `Body`（JSON 格式）和 `Form`/`File` 混合使用。

核心原因：因为一个 HTTP 请求的 **`Content-Type` 只能有一种**，要么是 JSON，要么是 Form。

#### 底层工作机制

> “**表单输入 ➔ 转换为 Python 变量 ➔ JSON 输出**”

1. **初始化启动，`File()` / `UploadFile` 包装声明**

   当启动 FastAPI 应用时，FastAPI 会**根据 `Form()` 或 `UploadFile` 类识别**表示**当前 API 接口预期要接收的 HTTP 请求体是一个 `Content-Type: multipart/form-data`  『纯文本+ 文件二进制流』的混合表单数据**。

2. **HTTP 请求解析**

   - 当**收到客户端发来的一个 HTTP 请求**时，FastAPI **识别检测**到携带的 **`Content-Type` 请求头字段**是 **`multipart/form-data`** 

   - FastAPI 会**借助 `python-multipart` **第三方底层依赖，**在 Request Body 请求体中**进行**解析**：

     ```ini
     POST /register4 HTTP/1.1
     Content-Length: 13480
     Content-Type: multipart/form-data;boundary=------FormBoundaryShouldDifferAtRuntime
     Host: 127.0.0.1:8000
     ------FormBoundaryShouldDifferAtRuntime
     Content-Disposition: form-data; name="avatar"; filename="1.png" # 根据 name 匹配 API 接口函数的形参变量名
     Content-Type: image/png # 根据这个识别到是一个文件流
     
     [message-part-body; type: "image/png", size: 12620 bytes]
     ------FormBoundaryShouldDifferAtRuntime
     Content-Disposition: form-data; name="username"
     
     jack
     ------FormBoundaryShouldDifferAtRuntime
     Content-Disposition: form-data; name="password"
     
     12345678
     ------FormBoundaryShouldDifferAtRuntime
     ```

     根据 **『`boundary=------xxxx` 随机分割字符串』**将 **“文本字段”** 和 **“二进制文件流”** 两者**剥离**分开：

     - **"文本数据"**：解析并**转换**为一个 **Python 字典**，并给 **API 接口函数中的 `Form()` 识别注入到同名的 `<形参变量>` 中**
     - **“文件二进制流”**：
       - 根据 **`Content-Type` 识别**到是一个 “文件二进制流数据”
       - 根据 **`Content-Disposition: form-data; name="avatar"`** 进行**键值对形式**的**临时存储**
         - 以**『 `name` 作为 Key，“文件二进制流数据” 作为 Value 值』**的形式**存储到临时内存**中

3. **`File()` / `UploadFile` 匹配并解析注入**

   - 当 `python-multipart` 解析处理完表单数据后，进入 API 接口函数中执行
   - FastAPI 识别到 **API 接口函数**中定义有 **`<形参变量>: UploadFile` / `<形参变量> = File()`** 时，会采用**命名匹配机制**，在内存中**查找**与**HTTP 表单请求体中与 `<形参变量名>` 同名的 `name` 键**
   - 当查找到与 `<形参变量名>` 的同名 `name` 后，FastAPI 会**将其对应的 Value 值（`文件二进制流数据`）注入到 `<形参变量>` 中**

4. **API 接口函数返回 JSON 格式的响应数据包**

   在 API 接口处理完逻辑返回响应后，**响应数据包**中的**`Content-Type` 响应头字段依然还是 `application/json`**，目的是为了**方便客户端以 JSON  的格式解析响应体**。

**设计哲学**：**“请求格式与响应格式完全解耦”**。

**核心机制（输入不等于输出）**：

​	客户端可以使用**表单（Form）**、**文件（File）** 甚至 **XML（自定义解析）** 各种奇形怪状的格式把数据送进来；但在经过 FastAPI 的统一洗礼后，进到你业务代码里的都是标准 Python 变量，而返回给前端的又是标准统一的 **JSON**。这种设计极大地降低了前后端的对接成本。

简单理解，可以把 `File()` 看作是**管道的入口标签**，把 `bytes` 或 `UploadFile` 看作是**管道末端的容器**。

### `File` 类（包装、校验）

#### 核心概念

核心功能：

- `File()` 本质上是一个**类函数**；
- 用于告诉 FastAPI **应该从 HTTP 请求的 `multipart/form-data` 表单请求体中去提取原始二进制流数据**
- 它**继承**了 FastAPI 底层的 **`FieldInfo` 元信息基类**，可以传入一些**值校验、元信息描述等参数配置**，可用于 Swagger 文档渲

简单来说，**`File()` 会接收一个原始二进制流数据**，并**提供**一些额外的**元信息描述**的**配置说明**，用于 **Swagger 文档渲染**。

#### 基本语法

FastAPI 支持通过在 API 接口函数中的**「形参列表」**中**定义一个或多个 `形参变量: bytes = File()`** 来表示**表单文件的解析接收**。

```python
from fastapi import FastAPI, File
from typing import Annotated

<FastAPI 应用实例> = FastAPI()

@<FastAPI 应用实例>.post('<URL 路径>')
async def <API 路由函数>(<形参变量x>: bytes = File()): pass


@<FastAPI 应用实例>.post('<URL 路径>')
async def <API 路由函数>(<形参变量x>: Annotated[bytes, File()]): pass
```

核心作用：让 FastAPI **从 HTTP 请求的 `multipart/form-data` 表单请求体中去提取原始二进制流数据（`bytes`）**。

核心特性：

- `File()` 会**一次性拿取所有**的**文件二进制字节流数据**，并**存入内存**中
- 后续再**以 Python 的 `bytes` 类型解析并注入到 `形参变量x` 中**
- `File()` 接收到的纯粹是**一串字节流**，所以适合**极小文件**的上传**接收**；*大文件用 `File()` 会导致**内存暴涨***

> 注：如果**不使用 `File()`**，FastAPI 会**默认**把 `<形参变量x>` 当作一个**Query 查询参数（Query Parameter）**。

##### 命名匹配机制

```python
<形参变量x>: bytes = File()
<形参变量x>: bytes | None = File(None) # 可选的文件
<形参变量x>: Annotated[bytes, File()]
<形参变量x>: Annotated[bytes | None, File()] = None # 可选的文件
```

FastAPI 规定，在 **API 接口函数**中定义的 **`<形参变量x>`** 必须**与 HTTP 请求中 Request Body 请求体的 FormData 表单数据字段的 `name` 属性同名**：

- **✅️同名**：FastAPI **正确匹配到 Body 请求体中对应的 FormData 表单数据字段的 `name` 属性名，解析注入**
- **❌️不同名**：FastAPI **无法匹配到对应的  Body 请求体中对应的 FormData 表单数据字段的 `name` 属性名**，抛出 **`422 Field required` 异常**

解决方案：可以使用 **`File()` 类的 `alias` 参数设定别名**，让 **`alias` 别名去匹配 FormData 表单数据字段的 `name` 属性名**，而 **API 接口函数中的 `形参变量` 则使用其他名称**。

#### 包装、校验

核心要点：`File()` 是一个集**位置声明（包装）、安全把关（参数值校验）**和**代码即文档（元数据描述）**于一体的生产力工具。

核心定义：在 FastAPI 中，**`File()` 函数用于显式声明一个形参应该从 HTTP 请求体（Body）的表单数据中提取解析**。它除了继承自 Pydantic 的基础校验参数（如 `default`, `description`）之外，还包含了一些专门用于控制**请求体解析行为**和 **OpenAPI/Swagger 文档生成**的高级参数。

`File` 类提供了如下校验方式：

```python
File(
	default: Any = Undefined, # 静态默认值
    default_factory: (() -> Any) | None = _Unset, # 动态默认值（工厂）
    media_type: str = "multipart/form-data", # 预期接收的数据格式（Content-Type）
    
    # 长度限制，适用于 str、list\dict\set\tuple 等序列容器数据类型
    min_length: int | None = None, # 最小长度
    max_length: int | None = None, # 最大长度
    
    # 数值范围验证，适用于 int、float 数值类型
	gt: float | None = None, # 大于 ＞
    ge: float | None = None, # 大于等于 ≥
    lt: float | None = None, # 小于 ＜
    le: float | None = None, # 小于等于 ≤
    
    # 正则表达式校验
    pattern: str | None = None,
    regex: str | None = None,
    
    # 其他（Swagger UI 文档标注）
    alias: str | None = None, # 别名
    title: str | None = None, # 参数标题
    description: str | None = None, # 参数描述信息
    deprecated: bool | None = None, # 是否已被废弃
    #...
)
```

##### 底层实现

- `File` 的底层实现本质上是**继承自 Pydantic 的 `Form`**，只不过是**将 `media_type` 改为了 `multipart/form-data`**。

```python
# FastAPI 源码核心逻辑简化
def File(
    default: Any = Undefined,
    *,
    media_type: str = "multipart/form-data", # 强制指定多部分表单类型
    **extra: Any,
) -> Any:
    return ParamTypes.File(  # 最终返回的是一个配置元数据对象
        default,
        media_type=media_type,
        **extra,
    )
```

#### 与 `Form()` 的混用

##### 抽象继承关系

关键点：**`File()` 是 继承自 `Form()` 的子类，而 `Form` 又是继承自 Body 的子类**。

> Body ──► Form ──► File

- **`Body`**：最基础的**元数据类**，让 FastAPI **“去 请求体（Request Body）里找这个数据”**，默认解析为 **`application/json`（JSON）**

- **`Form`**（继承自 `Body`）：修改了媒体类型（Media Type）

  ​    让 FastAPI **“去 请求体（Request Body）中的表单数据里找”**，解析为 **`application/x-www-form-urlencoded`（纯文本）** 

- **`File`**（继承自 `Form`）：进一步扩展了媒体类型，解析为 **`multipart/form-data`（纯文本字段 + 文件的二进制流）**

简单理解为：**表单数据**是**位于 Request Body 请求体中**的，而**文件上传**是**表单数据的扩展能力**，所以 **`Form` + `File`** 表示在**具有表单提交的基础能力之上**，**衍生**了可以**混合上传文件的功能**。

> [!IMPORTANT]
>
> 注意点：
>
> ​	因为 **`File`** 和 **`Form`** 都是**基于 `multipart/form-data` 工作**的，在**同一个路由函数**里，**不能**把 **`Body`（JSON 格式）和 `Form`/`File` 混合使用**。
>
> - 因为一个 HTTP 请求的 **`Content-Type` 只能有一种，要么是 JSON，要么是 FormData**。

##### 核心概念

从 HTTP 协议的角度来看：

- **单纯的 `Form`（普通表单）**只能**传输 `k=v` 键值对字符串**（比如用户名、密码）

- 当**需要上传包含二进制文件**（如图片、文档）的**复杂表单**时，单纯的字符串就无能为力了

这时候就轮到 **`multipart/form-data`**（也就是 `File` 所代表的格式）登场了。

**`multipart/form-data` 保留了表单可以同时提交多个键值对的特性**，但通过特定的“边界分隔符（Boundary）”，**允许表单中的某些字段包含复杂的二进制文件数据**。

所以，在 FastAPI 中会发现：

- API 接口函数中可以**同时存在声明**了： **`File(...)`** 的**文件二进制流数据**、 **`Form(...)`** 类型的**普通文本参数**
- 底层解析时，它们**共享同一套表单解析逻辑**

##### 示例

```ini
# HTTP 请求
POST /set_user_info HTTP/1.1
Content-Length: 470
Content-Type: multipart/form-data;boundary=------FormBoundaryShouldDifferAtRuntime
Host: 127.0.0.1:8000
------FormBoundaryShouldDifferAtRuntime
Content-Disposition: form-data; name="name"

jack
------FormBoundaryShouldDifferAtRuntime
Content-Disposition: form-data; name="age"

24
------FormBoundaryShouldDifferAtRuntime
Content-Disposition: form-data; name="sex"

男
------FormBoundaryShouldDifferAtRuntime
Content-Disposition: form-data; name="avatar_file"; filename=""
Content-Type: application/octet-stream


------FormBoundaryShouldDifferAtRuntime--
```

代码：

```python
from enum import Enum
from typing import Annotated
from fastapi import FastAPI, File, Form
from pydantic import Field

app = FastAPI()

class SexEnum(str, Enum):
    male = '男'
    female = '女'

# 混合表单（字段 + 文件上传）
@app.post("/set_user_info")
async def create_upload_file3(
    # 纯文本字段
    name: Annotated[str, Form(description='姓名')],  # 必填
    age: Annotated[int, Form(description='年龄', ge=18, le=100)],  # 必填
    sex: Annotated[str, Form(description='性别')] = SexEnum.male,  # 非必填

    *,  # 消除给定了默认值的参数只能写在最后一个位置的限制，* 后面的只能是关键字参数

    # 文件字段
    avatar_file: Annotated[bytes | None, File(description='上传的文件')]
):
    print(name)  # jack
    print(age)  # 19
    print(sex)  # 男
    print(avatar_file)  # b'80\x00\x01\x02\x04\x08\x10 @\x80\x001....'

    # 写入文件
    with open('avatar.png', 'wb') as f:
        f.write(avatar_file)

    return {'status': 'ok'}


# 启动应用
if __name__ == '__main__':
    import uvicorn

    uvicorn.run('server27:app', host="127.0.0.1", port=8000, reload=True)
```

##### 注意点（Pydantic 模型）

> - 在 FastAPI 中，`Form()` 或 `File()` 只能用于声明**独立的单个参数**，因此，**不能**在**同一个接口中同时使用 Pydantic 模型（JSON）和 `File`**，必须**拆分独立声明**。
>
> - 在最新版本的 FastAPI 中，使用 **`Form()` 包装的 Pydantic 模型**：
>
>   - **只能解析 `application/x-www-form-urlencoded` 纯文本表单**
>
>   - 对于 **`multipart/form-data` （纯文本 + 文件二进制流）**， Pydantic 数据模型是**无法直接解析**的
>
>     所以**当提交的表单存在文件上传**时，要么选择**平铺声明 `Form()` + `File()`**，要么选择**使用 `Depends(<解析类函数>)` 的形式（代码极其繁琐）**

### UploadFile 类型（解析、包装）

核心定义：**`UploadFile`** 是 `fastapi` 模块提供的一种**数据类型**。

#### 基本语法

FastAPI 支持通过在 API 接口函数中的**「形参列表」**中**定义一个或多个 `形参变量: UploadFile`** 来表示**表单文件的解析接收**。

```python
from fastapi import FastAPI, UploadFile
from typing import Annotated

<FastAPI 应用实例> = FastAPI()

# 不带元数据描述的
@<FastAPI 应用实例>.post('<URL 路径>')
async def <API 路由函数>(
    <形参变量x>: UploadFile, # 必选的文件
    <形参变量y>: UploadFile | None = None # 可选的文件
): pass

# 带元数据描述的（File()）
@<FastAPI 应用实例>.post('<URL 路径>')
async def <API 路由函数>(
    <形参变量x>: Annotated[UploadFile, File(...)] # 必选的文件
    <形参变量y>: Annotated[UploadFile | None, File(...)] = None # 可选的文件
): pass
```

核心作用：

​	当一个变量被**声明为 `:UploadFile` 类型**时，表示要**接收一个原始二进制流的文件数据**，并**解析包装**为**一个 Python 标准的 File 文件对象**；且额外还**提供**了类似 Python 标准文件对象的 API（`filename` 文件名称、`size` 文件大小...）。

- 命名匹配机制：**`<形参变量x>: UploadFile`** 必须与 Body 请求体中**表单文件字段**的 **`name` 属性同名**，否则无法正确匹配。

> [!NOTE]
>
> **为什么 `<形参变量x> : UploadFile` 不会被误认为一个Query 查询参数？**
>
> ​	FastAPI 的源码内部有一个明确的判断分支：**只要识别到 `UploadFile` 类型，就会自动将其强制归类为 `Form/File` 参数（即 `multipart/form-data` 的 Body 参数）**。

##### 分块传输机制

核心特性：`UploadFile` 用于**大文件上传**，采用**分块传输二进制流**机制，一旦**超过分块传输的长度限制**则会**先写入硬盘的临时内存**中，**防止内存溢出**。

核心机制：`UploadFile` 内部维护着一个**内存阈值（默认 1MB）**，文件**若小于 10MB 则存储在内存**中，**一旦超过 10MB，它会自动分块写入硬盘的临时文件中**，从而保护内存。

```
客户端请求 (multipart/form-data)
      │
      ├──> 普通文本字段 (username=Admin) ──> 放入内存 ──> 转化为 Python 基础类型 (str, int)
      │
      └──> 文件字段 (avatar.png) 
                │
                ├──> 文件 < 10MB  ──> 留在内存 (UploadFile 内部)
                └──> 文件 > 10MB  ──> 分块写入硬盘临时文件 (UploadFile 内部)
```

#### 与 `File()` 的协作声明

在 FastAPI 中：

- `file: UploadFile` 和 `file: UploadFile = File(...)`（或者 `File(None)` / `File(default)`）

  核心区别在于：**显式声明（Explicit）与隐式推断（Implicit），以及如何定义默认值和元数据。**

| **特性**             | **file: UploadFile**             | **file: UploadFile = File(...)**                   |
| -------------------- | -------------------------------- | -------------------------------------------------- |
| **定义方式**         | **隐式**推断（Implicit）         | **显式**声明（Explicit）                           |
| **参数必填性**       | 默认**必填**                     | 取决于括号内的值（`...` 表示必填）                 |
| **支持元数据**       | 不支持（无法添加描述、别名等）   | **支持**（可以通过 `File()` 添加额外的验证或说明） |
| **FastAPI 内部处理** | 自动识别为 `multipart/form-data` | 明确指定为 `multipart/form-data`                   |

##### file: UploadFile 隐式

*优先使用*

这是 FastAPI 的一种**语法糖**。因为 `UploadFile` 只能来自于表单文件上传，FastAPI 会自动在后台将其视为 `File()` 参数

- **优点**：代码极其简洁
- **缺点**：**无法为这个文件字段添加更多的配置**（比如限制文件名、添加 Swagger 文档描述等）
- **适用场景**：只需要简单地接收一个**必填**文件，**不需要对 Swagger API 接口文档做特殊定制**，直接写 `:UploadFile` 最简单直接

```python
@app.post("/upload")
async def upload(file: UploadFile):  # 简单、直观，默认必填
    return {"filename": file.filename}
```

##### file: UploadFile = File() 显式

这里的 `File(...)` 是一个函数（实际上是 `Form` 的子类）。`...`（Python 的 Ellipsis 对象）在这里表示**该参数是必填的**。

- **适用场景**：当需要给这个文件参数**在 Swagger 文档里加描述（`description`）、别名（`alias`）或者默认值**时，才需要显式写出 **`= File(...)`**

```python
file: UploadFile = File(
    ..., 
    description="请上传您的个人头像（仅限 JPG/PNG）",
    alias="user_avatar"  # 前端上传时使用的字段名
), # 必填

file2: UploadFile | None = File(None) # 非必填
```

##### 综合总结

| **声明方式**                                     | **底层处理机制**                                             | **适用场景**                                                 | **优缺点**                                                   |
| ------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`file: bytes = File()`**                       | 整个文件一次性完整读入内存。                                 | **极小的文件**（如：头像、小图标、纯文本配置）。             | **优**：操作简单，直接拿到二进制。 **缺**：大文件会导致服务器内存直接爆满。 |
| **`file: UploadFile`** *(最常用)*                | 默认带有 `File()` 的隐式效果。小于 1MB 在内存，大于 1MB 自动写盘。提供 `file.file.read()` 等类文件操作。 | **绝大多数场景**、大文件、未知大小的文件（如：视频、大 PDF、压缩包）。 | **优**：安全、省内存，自带文件名和大小元数据。 **缺**：读取内容时需要使用 `await file.read()` 异步操作。 |
| **`file: UploadFile = File(description="...")`** | 机制同上，只是显式调用了 `File()` 来为 Swagger 文档或 Pydantic 校验增加额外元数据。 | 需要在 **Swagger UI 中写注释、定义限制** 的大文件上传。      | **优**：完美兼顾了大文件安全性和文档的高可读性。             |

### 文件流操作（异步）

#### 前置概念

核心概念：`UploadFile` 对象基本围绕**临时文件（暂存区）**的**读取、写入、移动文件流指针、关闭临时文件资源** 这 4 个基本操作。

##### 临时文件（暂存区）

核心定义：当**客户端（浏览器或手机 App）向服务器上传一个文件**时，**数据**是**通过网络“一长串字节流”源源不断发过来**的。

- 如果**直接放内存**：若**文件过大**会导致**服务器内存挤爆**，直接导致宕机
- 如果**直接写到目标文件**：在**写入过程中**，可能会出现**网络中断**等问题而**导致写入的文件数据是不完整的**，从而**在本地目录中留下了一个损坏的垃圾文件**

为了解决以上问题的出现，**FastAPI（底层通过 `Starlette` 和 `tempfile` 模块）会有一个判定**：

- 如果**文件很小**（比如小于 10MB）：直接**存入内存中**
- 如果**文件很大**：FastAPI 会在**本地系统的临时目录（`C:\User\AppData\Local\Temp`）**中创建一个**临时文件（暂存区）**，用于**保存接收到网络二进制流数据**

```
					 ┌─── 小于 10MB ───► 纯内存存储 (BytesIO)
                     │
[网络上传的二进制流] ───┤
                     │
                     └─── 大于 10MB ───► 自动硬盘暂存 (C:\...\Temp)
```

###### 核心结论

- **临时文件（暂存区）**是服务器为了**保护内存安全**、**确保文件完整性**而开辟的一块“中转站”
- **临时文件（暂存区）**并不是一个真实永久的文件，当**程序结束或手动调用 `file.close()` 方法**后，会被**彻底删除**，所以**需要将其写入真实物理位置的「目标永久文件」中**

###### 完整流程

可以把 FastAPI 的文件操作分为 2 个阶段（**临时文件、永久文件**）：

- **【阶段一：网络接收与暂存（FastAPI 自动完成）】**

  > **网络原始二进制流 → 拆分成 Chunk（分块） → FastAPI 内部执行 `write()` → 写入到系统临时文件（暂存区）**

- **【阶段二：业务代码处理】**

  - **临时文件（暂存区）的读写**：

    > - **写入**：**`UploadFile.write()` <追加或局部覆盖> 写入 → 系统临时文件（暂存区）**
    > - **读取**：**系统临时文件（暂存区）→  预读到内存的读缓冲区（BufferReader） → `UploadFile.read()` 分块读取 **

  - **真实永久文件的写入：**

    > - **写入**：**`UploadFile.read()` 分块读取  → `aiofiles.write()` 写入 → 真实物理位置的永久文件**

##### 文件流与文件指针

- **文件流**：是数据在程序和文件之间传输的**一种连续的、有顺序的数据传输通道（或序列）**（水管）
- **文件指针**：是文件流通道中的**一个位置标记，用来指出当前读写操作进行到了文件的哪个字节**（阀门）

核心定义：当一个**程序操作文件**时，会**开辟**一个**文件流空间**，在其中通过**「文件指针」移动位置**对**临时文件（暂存区）**进行**读写颗粒度**操作。

#### 核心方法

核心要点：**`UploadFile`** 对象的所有方法基本都是**『异步非阻塞』**的**协程函数**，所以需要**配合 `await` 关键字使用**。

##### read(size) 读取

核心描述：FastAPI 会**先从临时文件（暂存区）**中，**一次性「预读部分数据」到内存中的读缓冲区（BufferReader）**中，再**由 `read(size)` 分块流式读取文件数据（二进制字节流）**，**每次**仅**读取 10MB（`size`）** 的数据，避免一次性将大文件全部读入内存。

- 返回值：每次**读取并返回一串 `size` 大小的 `bytes` 原始二进制流文件数据（`chunk`）**。

###### size 参数

- **每次 `file.read()` 分块传输**时的**可读取的 `chunk` 最大字节数（阈值）**，*默认是 10MB*。

- 当**每次读取**的文件二进制流数据**超过 10MB 大小**时，会**一次性清空内部读缓冲区（BufferReader）临时存储的数据**，并**排出**返回给外部

  > 其实是 FastAPI 内部 **`SpooledTemporaryFile` 原始二进制流文件对象**的参数。当上传的文件小于 10MB 时，FastAPI 会把文件存放在**内存**中；一旦超过 10MB，它会自动将超出部分写入到系统的**磁盘临时文件**中。

可直接**传入 `file.read(size=1024*1024)` **用于设置**每次 `file.read()` 最大可读取的字节数**：

```python
content = file.read(1024*1024) # content 每次仅获取 1MB 的文件二进制数据，可传入给 file.write(data: bytes) 写入
```

> 注：
>
> - 如果**上传的文件总大小 < `read()` 每次可读取的最大字节数 `size`**， 则**当前读取并返回的 `chunk` 就是一整个文件**
> - 如果**上传的文件总大小 > `read()` 每次可读取的最大字节数 `size`**， 则**当前读取并返回的 `chunk` 就是文件的一部分**

****

###### 核心特性

- **`file.read()`** 是一个**『异步非阻塞』**的**协程函数**，当**线程遇到 `await`** 会**暂停**当前行位置**后续的代码执行**，**CPU 控制权交还给事件循环Event Loop**，并让**线程去执行其他 Task 任务（异步高并发）**

- 此时，`file.read()` 会持续在**从读缓冲区（BufferReader）中读取文件数据（`bytes`）**

- 在**每次读取的「二进制字节数」达到 `size`（阈值）**后，会**主动通知**事件循环，**让线程回到当前行位置继续执行**，它会 **`yield` 一次性排出读取到 `chunk` 数据，且内部读缓冲区（Buffer）会被消耗完**，此时**『文件流指针』会移动到下一块位置**

  整个过程类似于 **yield 生成器的 `next()`**

```python
[ 当前 Task ] -> 遇到 await file.read(size) -> 挂起
                                                   |
                                           (任务扔给线程池)
                                                   |
[ 事件循环 ] <--- 释放 CPU 控制权，去执行其他 Task <--+
     |
     | (当子线程读满 size 或到 EOF)
     v
[ 事件循环 ] ---> 收到完成通知 -> 恢复当前 Task -> 推进文件指针 -> 拿到 Chunk 数据
```

###### 示例

针对这一特性，为了**保证能完整读取完文件的所有数据**，通常需要**配合 `while` 循环来重复读取下一个 `chunk` 文件分块数据**：

> 这会涉及到**双重 `await file.read()`**：
>
> - 第一个 `file.read()`：第一次读取出 1MB 的文件数据，给 `while` 循环内的 `file.write()` 写入，**防止没有内容写入而报错**
>
> - 后续 `while` 循环中的 `file.read()`：
>
>   ​	由于**内部读缓冲区（Buffer）已被清空**，开始**读取下一个 `chunk` 分块（通过文件流指针）**，并返回给 `content`，调用 `file.write()` 写入当前读取返回的 `chunk` 文件数据；
>
>   持续不断**重复执行**这一流程，直到整个文件数据被读取完（内部返回 *`StopIteration` 异常*）

```python
import aiofiles # 支持异步的文件操作库，可使用 async with（异步上下文管理器协议）

@app.post("/uploadfile")
async def create_upload_file(file: UploadFile = File(..., description="上传文件")):
    # Python 原生的 open() 是同步阻塞的，在异步路由中直接使用会阻塞事件循环。所以需使用 aiofiles 异步流式写入。
    CHUNK_SIZE = 1024 * 1024  # 1MB

    async with aiofiles.open(f"_xw_{file.filename}", 'wb') as f:
        while True:
            # 每次读取 1MB 字节，避免一次性将大文件全部读入内存，读取完返回数据之后，便会清空内部缓冲区
            # 随着文件指针的向后移动，需要重复循环异步读取下一块文件数据
            # 当达到 1MB 字节（阈值）后，向事件循环（EventLoop）发送报告，告知线程可以回来继续执行，将数据返回给 content
            content = await file.read(CHUNK_SIZE)
            if not content:  # 读到文件末尾 (EOF)，返回 b''
                break
            await f.write(content)
		
        # while content := await file.read(CHUNK_SIZE) # 简写语法
            
	# 离开 async with，自动执行 file.close() 关闭临时文件资源

    """
    【执行流程解析】
    1. 线程（事件循环）执行到 `await file.read(1024*1024)` 时会暂停当前协程，优先去处理其他并发请求。
    2. 当底层成功读取到 1MB 字节 的数据后，协程被唤醒，将数据赋值给 `content` 并进入 while 循环。
    3. 遇到 `await f.write(content)` 时再次暂停，等待底层将这 1MB 字节数据异步写入磁盘，写入完成后继续向下执行。
    4. 核心原理：`UploadFile` 的底层读写是流式的。每次调用 `file.read(size)` 就像调用生成器的 `next()` 一样，
       指针向后移动。直到读完文件最后一部分，`file.read()` 会返回空字节串（b''），此时 `while content` 
       条件不成立，循环结束，异步文件上下文关闭。
    """

    return {'status': 'ok'}
```

##### write(bytes) 写入

核心定义：在 FastAPI 打开的**文件流**中，通过**文件指针**对**临时文件（暂存区）【当前上传的文件】**进行**追加写入或覆盖操作**。

**临时文件**：可以理解为**还未被写入到真实物理位置的一个虚拟临时文件**，在程序结束后会被彻底删除，后续可**通过其他异步 I/O 库（`aiofiles` ）写入到目标文件中持久化存储**。

参数：

- **`bytes`：将数据以一个个 byte 字节的形式，写入到当前操作的临时文件（暂存区）中**。

注：**`UploadFile.write()` 只负责对临时文件的修改，需通过 `aiofile.open()` 方法来将「临时文件的内容」写入到「真实物理永久文件」中**。

```python
# 实现对当前上传的文件内容（临时文件）添加一个 “前缀” 内容，并把修改后的临时文件写入到真实物理位置的永久文件中。
import asyncio

@app.post('/upload_file3')
async def upload_file3(file: UploadFile = File(..., description="上传文件")):
    # 1. FastAPI 会默认打开一个文件流空间，用于 read/write 读取/写入临时文件，通过 seek() 操作文件指针

    # 2. 从临时文件（暂存区）中读取文件数据，放入读缓冲区（Buffer）中
    origin_content = await file.read()
    print(origin_content)  # 修改之前的临时文件内容：b'Hello World'

    await file.seek(0)  # 3. 由于 file.read() 读取了文件数据，所以需要将文件指针重新指向文件开头

    await file.write(b'Prefix:' + origin_content)  # 4. 向临时文件（暂存区）写入新的前缀 + 原内容

    await file.seek(0)  # 5. 重新将文件指针指向文件开头

    new_content = await file.read()  # 6. 再次重头开始读取临时文件（暂存区）的内容
    print(new_content)  # 修改之后的临时文件内容：b'Prefix:Hello World'

    # 7. 通过 aiofiles 模块，将『临时文件（暂存区）』的内容写入到「本地永久文件」中
    async with aiofiles.open(f'./_new_{file.filename}', 'wb') as f:
        await f.write(new_content)

    await file.close()  # 8. 异步关闭临时文件（暂存区），即时释放内存和临时文件资源的引用

    return {'status': 'ok'}
```

##### seek(offset) 移动文件流指针

核心定义：在 **`read()` 读取 → `write()` 写入**这个过程中，是在**文件操作流空间内执行**的，通过**「文件指针」一步步往下移动位置来读取/写入**，每次移动**以字节（byte）为单位**，**对流式数据**进行**精确的重组和操控**。

核心功能：可以**在文件流中，调用  `file.seek()` 操作文件指针**以**向前、向后移动位置**的形式，**对正在传输的「当前 `chunk` 文件分块数据」进行操作**，可实现**覆盖、重写、添加额外内容** 等操作，以**字节（`byte`） 为单位**进行**颗粒度控制**。

<img src="C:/Users/win/Desktop/学习笔记体系/AI Agent/Python/03-Python Web 框架/FastAPI 异步框架/media/UploadFile-file.seek() 文件流操作与指针控制.png" style="zoom:60%;" />

###### offset 参数

核心作用：在当前操作的 `chunk` 分块文件流中，**每次文件指针移动的 `byte` 字节数（位置）**

| **基准值 **        | **含义**                              | **常见应用场景**                                             |
| ------------------ | ------------------------------------- | ------------------------------------------------------------ |
| **`0` (文件开头)** | 从文件绝对位置 `0` 字节开始计算偏移量 | **重头开始重写文件**                                         |
| **`1` (当前位置)** | 从当前指针所在的位置开始**相对移动**  | 在当前 `chunk` 中，需要**“向前倒退”**几个字节重写，或者**向后跳过**一段无效数据 |
| **`2` (文件末尾)** | 从文件的最末尾开始**倒序计算偏移量**  | **断点续传**或**追加数据**。比如获取已下载文件的末尾位置，继续写入下一个 `chunk` |

> 注：
>
> - 若**文件总大小 < `read()` 每次可读取的最大字节数 `size`**， 则**当前 `seek()` 文件指针操作的 `chunk` 就是一整个文件**
> - 若**文件总大小 > `read()` 每次可读取的最大字节数 `size`**， 则**当前 `seek()` 文件指针操作的 `chunk` 就是文件的一部分**

###### 示例

```python
import asyncio

# 给上传的文件加一个 “前缀”，并写入到本地硬盘中
@app.post('/upload_file2')
async def upload_file2(file: UploadFile = File(..., description="上传文件")):
    print(file.filename) # test.md

    # 1.异步分块流式读取文件数据（二进制字节流），每次仅读取 10MB 的数据，放入缓冲区中，避免一次性将大文件全部读入内存。
    content = await file.read()  # 假设 read() 读取到的已经是一整个文件（文件总大小 < size）

    print(content)  # b'80\x00\x01\x02\x04\x08\x10 @\x80\x001....'
    await file.seek(0)  # 2.将文件操作流中的文件指针移动到 `chunk` 文件分块开头，准备覆盖或重写

    # 4. 打开本地文件，将 read() 读取到的 chunk 文件数据（位于暂存区内）通过 write() 写入到本地硬盘中
    async with aiofiles.open(f'./{file.filename}', 'wb') as f:
        # 5. 将读取存入的临时文件（暂存区），异步写入新的前缀 + 原内容
        log_prefix = b'[Process at unviron] '
        await f.write(log_prefix + content)

    await file.close()  # 6. 异步关闭文件资源，即时释放内存和临时文件资源的引用

    return {'status': 'ok'}
```

##### close() 关闭文件流

虽然 FastAPI 会在请求结束时**自动**关闭文件，但在处理**大文件**或**高并发**时，显式地手动调用 `await file.close()` 是非常重要的性能优化手段。

核心作用：当文件处理完后，调用`close` 可以**即时释放内存和系统临时文件的引用**。

核心特性：

- 由于 **FastAPI 会在本地系统目录下创建一个临时文件（暂存区）的引用**；
- 当文件处理完后，如果**不手动 `close`**，这个**系统临时文件会一直存在消耗句柄引用**，直到**整个 HTTP 请求完全结束、响应返回给用户之后**才会被**释放**，从而导致**不必要的系统资源消耗**

```python
import asyncio

@app.post("/heavy-upload/")
async def heavy_upload(file: UploadFile):
    # 1. 把文件保存到本地目录
    async with asyncio.open(f"./save/{file.filename}", "wb") as f:
        content = await file.read()
        f.write(content)
    
    # 2. 核心：立即手动关闭并删除临时文件！
    # 这样可以立刻释放磁盘空间，不用等下面耗时的后续逻辑跑完
    await file.close() 
    
    # 3. 假设后面还有一段很耗时的业务逻辑（比如发送邮件通知、分析数据等）
    await asyncio.sleep(10) 
    
    return {"status": "success"}
```

如果不加 `await file.close()`，在第 3 步那 10 秒钟内，**系统的临时文件会一直被占用**。如果高并发场景下有 1000 个人上传大文件，服务器的硬盘可能瞬间就被临时文件撑爆了。

##### List[UploadFile] 上传多个文件

核心描述：如果需要**一次性上传多个文件**，只需要将**类型声明为 `List[UploadFile]` \  `list[UploadFile]`** 即可。

```python
import aiofiles
from datetime import datetime
    
# 上传多个文件
@app.post('/upload_files')
async def upload_files(
    filetag: Annotated[str, Form()],
    datetime: Annotated[datetime, Form()],
    files: Annotated[list[UploadFile], File(..., description='上传的文件列表')]
):
    print(filetag) # tag
    print(datetime) # 2022-10-26 00:00:00
    
    print(files)  # UploadFile 文件列表
    # [UploadFile(filename='5658.jpg', size=102542, headers=Headers({'content-disposition': 'form-data; name="files"; filename="5658.jpg"', 'content-type': 'image/jpeg'})), UploadFile(filename='18741.png', size=578036, headers=Headers({'content-disposition': 'form-data; name="files"; filename="18741.png"', 'content-type': 'image/png'}))]

    for file in files:
        print(file.filename)
        print(file.headers)
        print(file.size)  # 文件的大小

        CHUNK_SIZE = 1024 * 1024

        # 依次将每个文件写入到本地硬盘中
        async with aiofiles.open(f"./_adw_{file.filename}", 'wb') as f:
            while True:
                content = await file.read(CHUNK_SIZE)
                if not content:
                    break
                await f.write(content)

    return {'status': 'ok'}
```

###### form() 获取表单数据

核心作用：`Request` 对象**自动将原始的二进制字节码请求体解析为一个 `FormData` 表单对象并返回**，并**异步**获取。

> 注：要求 Content-Type 为 **`multipart/form-data`**。

关键点：需额外引入 **`fastapi.datastructures.FormData` 类型**来方便类型提示。

- **普通表单数据**：

  ```ini
  # HTTP 请求
  POST /get_from_data HTTP/1.1
  Content-Length: 385
  Content-Type: multipart/form-data;boundary=------FormBoundaryShouldDifferAtRuntime # 数据传输格式
  Host: 127.0.0.1:8000
  
  # 请求体
  ------FormBoundaryShouldDifferAtRuntime
  Content-Disposition: form-data; name="account"
  
  jack
  ------FormBoundaryShouldDifferAtRuntime
  Content-Disposition: form-data; name="password"
  
  123456
  ------FormBoundaryShouldDifferAtRuntime--
  ```

  代码示例：

  ```python
  from fastapi.datastructures import FormData
  from fastapi import FastAPI, Request
  
  app = FastAPI()
  
  # 普通 FormData 表单数据
  @app.post('/get_from_data')
  async def get_from_data(request: Request):
      # 等待获取请求体中的 FormData 表单数据
      form_data: FormData = await request.form()
  
      print(form_data)  # FormData([('account', 'jack'), ('password', '123456')])
  
      # 获取 Body 请求体 的 Key-Value 键值对
      for key, value in form_data.items():
          print(f"{key}: {value}")
  
      '''
      account: jack
      password: 123456
      '''
  
      # get(key=): 根据 key 获取指定 Body 请求体字段的 value 值
      print(form_data.get('account'))  # jack
      print(form_data.get('password'))  # 123456
      ...
  ```
